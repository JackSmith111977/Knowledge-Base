## 1. 概述与背景

### 1.1 ADK 是什么

**定义**：Agent Development Kit (ADK) 是 Google 推出的模块化、框架无关的 AI 智能体开发工具包，旨在使智能体开发感觉更像传统软件开发。

**核心理念**：
- **模型无关性**：虽然针对 Gemini 和 Google 生态系统优化，但 ADK 不绑定特定模型
- **部署无关性**：支持本地、云端、混合部署等多种部署方式
- **兼容性优先**：构建为与其他框架兼容，非私有封闭生态
- **开发体验**：让创建、部署和编排智能体架构感觉直观

**官方资源**：
- 官方文档：https://google.github.io/adk-docs/
- GitHub 仓库：https://github.com/google/adk-python

### 1.2 SKILL.md 规范简介

**起源**：Agent Skills 最初由 Anthropic 于 2025 年 10 月发布，随后作为开放标准正式发布（https://agentskills.io/specification），成为跨平台通用规范。

**文件结构**：
```
my-skill/
├── SKILL.md          # 必需：YAML frontmatter + Markdown 指令
├── scripts/          # 可选：可执行脚本
├── references/       # 可选：参考文档、规范标准
└── assets/           # 可选：模板、资源文件
```

**SKILL.md 组成**：
1. **YAML Frontmatter**：元数据定义（name、description 等必需字段）
2. **Markdown 正文**：详细的使用说明、执行步骤、注意事项

**核心设计哲学——渐进式加载**：
| 阶段 | 加载内容 | Token 消耗 |
|------|----------|------------|
| 启动时 | name + description | ~100 tokens |
| 触发时 | 完整 SKILL.md 内容 | < 5000 tokens |
| 执行时 | scripts/、references/ 等资源 | 按需加载 |

这种设计确保上下文不会爆炸，同时保证 Agent 能正确发现和调用 Skill。

### 1.3 为什么需要设计模式

**从格式到内容的转变**

随着 Agent Skills 的普及，开发者面临一个普遍困境：**SKILL.md 写对了，但不好用**。问题的根源在于将传统 Prompt 思维直接套用到 Skill 设计中，缺乏对 Agent 行为模式的系统性理解。

**5 种模式解决的问题域**

| 模式 | 问题域 | 核心价值 |
|------|--------|----------|
| **Tool Wrapper** | 技术栈规范塞入 system prompt 导致上下文膨胀 | 按需加载，规范独立维护 |
| **Generator** | 输出格式不一致、结构混乱 | 模板填充，确保可预测性 |
| **Reviewer** | 审查标准写死难以更新、跨项目复用难 | 流程与标准分离，独立版本控制 |
| **Inversion** | 用户输入不完整导致返工率高达 75% | 主动采访确认，硬性门控禁出 |
| **Pipeline** | Agent 跳过设计直接写代码 | 强制顺序执行，检查点门控 |

## 2. 模式 1: Tool Wrapper (工具封装器)

### 2.1 核心思想

**工具封装器 (Tool Wrapper)** 是 5 种模式中最基础、最容易上手的一种。它的核心思想可以用一句话概括：

> **按需加载上下文，不做"无用功"**

**问题背景**：
很多开发者在设计 Agent Skill 时，容易犯一个常见错误：把所有相关的知识、规范、API 定义都一股脑写进系统提示词 (system prompt) 中，不管 Agent 当前任务是否用得到。这种做法导致：
- **上下文膨胀**：大量无关知识长期占用宝贵的 token 配额
- **性能下降**：模型需要处理更多噪声，响应速度变慢
- **维护困难**：每次更新规范都要修改系统提示词，影响所有任务

**Tool Wrapper 的解决方案**：
将特定技术栈、框架、库的规范文档打包成一个独立的 Skill，Agent 只在真正需要使用该技术时，才动态加载对应的上下文。这种**按需注入**的机制，让 Agent 保持轻量的同时，又能在需要时"瞬间成为专家"。

**来源**：Google ADK 团队研究 Anthropic、Vercel 等公司的 Skill 实践后发现，这是最基础也最常用的模式。

### 2.2 工作机制

Tool Wrapper 模式的工作流程遵循**渐进式披露**原则：

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 1: 启动时                                              │
│  Agent 仅加载 SKILL.md 的 YAML frontmatter                  │
│  (name + description, 约 100 tokens)                         │
│  ↓ 判断用户请求是否匹配技能描述                              │
├─────────────────────────────────────────────────────────────┤
│  阶段 2: 触发时                                              │
│  用户请求命中关键词 → 加载完整 SKILL.md 指令 (< 5000 tokens) │
│  ↓ 根据指令判断是否需要加载参考资料                          │
├─────────────────────────────────────────────────────────────┤
│  阶段 3: 执行时                                              │
│  按需加载 references/ 目录下的规范文档                       │
│  将规范作为"绝对真理"应用于任务                              │
└─────────────────────────────────────────────────────────────┘
```

**关键技术点**：
1. **关键词监听**：SKILL.md 的 description 字段需要精心设计，包含所有可能的触发关键词
2. **条件加载**：指令中明确说明"仅在审查或编写代码时加载 conventions.md"
3. **规则优先级**：加载的规范文档被视为该场景下的最高优先级规则

### 2.3 完整 SKILL.md 代码示例

以下是 FastAPI 开发专家的 Tool Wrapper 示例：

```yaml
# skills/fastapi-expert/SKILL.md
---
name: fastapi-expert
description: FastAPI 开发最佳实践与规范。在构建、审查或调试 FastAPI 应用、REST API、WebSocket、Pydantic 模型、依赖注入时使用。
metadata:
  pattern: tool-wrapper
  domain: fastapi
  version: 1.0.0
---

# FastAPI 开发专家

你是 FastAPI 开发专家。将以下规范应用于用户的代码或问题。

## 核心规范

**仅在开始审查或编写 FastAPI 代码时**，加载 `references/conventions.md` 获取完整的 FastAPI 最佳实践列表。

## 审查代码时

1. **加载规范参考文件** - 执行前先加载 `references/conventions.md`
2. **逐条对照检查** - 根据规范中的每条规则检查用户代码
3. **违规报告** - 对于每处违规：
   - 引用具体规则编号和原文
   - 指出代码中的具体问题位置
   - 给出符合规范的修复建议代码

## 编写代码时

1. **加载规范参考文件** - 开始编码前必须先加载 `references/conventions.md`
2. **严格遵循规范** - 将规范中的每条规则视为必须遵守的约束
3. **类型注解要求**：
   - 为所有函数签名添加类型注解 (包括参数和返回值)
   - 使用 Python 3.10+ 的类型语法 (如 `str | None` 而非 `Optional[str]`)
4. **依赖注入风格**：
   - 使用 `Annotated` 风格进行依赖注入
   - 示例：`async def get_user(user_id: Annotated[int, Path(gt=0)])`

## 调试问题时

1. 根据错误类型判断是否涉及 FastAPI 规范
2. 如涉及，加载规范参考文件排查
3. 给出符合规范的解决方案

## 禁止行为

- ❌ 不要在没有加载规范的情况下给出 FastAPI 代码建议
- ❌ 不要使用过时的 FastAPI 语法 (如旧的依赖注入方式)
- ❌ 不要忽略类型注解要求
```

### 2.5 适用场景

Tool Wrapper 模式特别适合以下场景：

| 场景 | 说明 | 典型例子 |
|------|------|----------|
| **团队编码规范分发** | 将团队内部的编码标准打包成 Skill | Python 风格指南、Git 提交规范 |
| **框架最佳实践注入** | 特定框架的使用规范和最佳实践 | React、Vue、FastAPI、Django |
| **内部知识库沉淀** | 将公司内部系统的使用文档打包 | 内部 API 文档、微服务调用规范 |
| **库/SDK 使用指南** | 第三方库的正确使用方式 | Stripe SDK、Supabase 客户端 |
| **安全合规要求** | 安全编码标准和合规检查 | OWASP Top 10、GDPR 合规指南 |

**核心价值**：
- 让 Agent 即时掌握任意库的规范
- 无需将 API 约定硬编码到系统提示词中
- 规范独立维护，版本控制更清晰

### 2.6 注意事项

**1. description 字段设计**
- description 是 Agent 判断是否触发 Skill 的唯一依据
- 必须包含所有可能的触发关键词
- 示例：`description: FastAPI 开发最佳实践。在构建、审查、调试 FastAPI 应用、REST API、WebSocket、Pydantic 模型时使用`

**2. 条件加载指令**
- 明确说明何时加载参考资料："仅在开始审查或编写代码时加载"
- 避免无条件加载所有参考文件，防止上下文膨胀

**3. 规则优先级说明**
- 明确说明加载的规范是"绝对真理"，必须严格执行
- 如有多条规范冲突，说明优先级顺序

**4. 文件命名规范**
- references/ 目录下的文件使用小写字母和连字符
- 文件命名要有语义，如 `conventions.md` 而非 `rules.md`

**常见错误**：
- ❌ description 太简短，Agent 无法正确触发
- ❌ 无条件加载所有参考文件，浪费 token
- ❌ 规范文件内容过时，与实际要求不符
- ❌ 缺少"禁止行为"说明，Agent 可能违反规范

## 3. 模式 2: Generator (生成器)

### 3.1 核心思想

**生成器 (Generator)** 模式的核心思想：

> **模板 + 风格指南固化结构，消除输出的随机性**

**问题背景**：
使用 Agent 生成文档时，开发者经常遇到以下问题：
- **结构不一致**：同一类文档，每次生成的结构都不同
- **格式混乱**：标题层级、术语使用、代码块格式不统一
- **内容缺失**：关键章节时有时无，取决于 Agent 的"心情"
- **风格漂移**：语气时正式时随意，术语前后不一致

这些问题在工程化场景中是不可接受的。例如：
- API 文档需要统一的接口说明、参数表、返回值、错误码结构
- Commit 消息需要遵循团队约定的格式 (如 Conventional Commits)
- 技术报告需要固定的章节安排和呈现方式

**Generator 的解决方案**：
将输出结构固化为模板 (assets/)，将格式规则固化为风格指南 (references/)，Agent 扮演"模板填充器"而非"自由创作者"的角色。SKILL.md 作为项目经理，协调整个生成流程。

### 3.2 工作机制

Generator 模式的工作流程：

```
┌─────────────────────────────────────────────────────────────┐
│  步骤 1: 加载风格指南                                         │
│  读取 references/style-guide.md                               │
│  获取语气、格式、术语规范                                     │
├─────────────────────────────────────────────────────────────┤
│  步骤 2: 加载输出模板                                         │
│  读取 assets/template.md                                      │
│  获取所需的输出结构和必需字段                                 │
├─────────────────────────────────────────────────────────────┤
│  步骤 3: 采访用户收集变量                                     │
│  识别模板中需要用户提供的变量                                 │
│  主动询问缺失的信息                                           │
├─────────────────────────────────────────────────────────────┤
│  步骤 4: 填充模板                                             │
│  按照风格指南的规则                                           │
│  将用户提供的变量填入模板                                     │
│  确保模板每个部分都出现在输出中                               │
├─────────────────────────────────────────────────────────────┤
│  步骤 5: 返回完整文档                                         │
│  输出结构化的 Markdown 文档                                   │
│  格式、风格完全符合规范                                       │
└─────────────────────────────────────────────────────────────┘
```

**关键组件**：
- **assets/**: 存放输出模板，定义 "What to produce"(必须包含哪些部分)
- **references/**: 存放风格指南，定义 "How to write"(如何书写每个部分)
- **SKILL.md**: 扮演项目经理角色，协调执行顺序

### 3.3 完整 SKILL.md 代码示例

以下是技术报告生成器的 Generator 示例：

```yaml

---
name: tech-report-generator
description: 从模板生成标准化技术报告。在需要生成 API 文档、项目周报、技术方案、评审报告等结构化文档时使用。

# 技术报告生成器

你是一个技术报告生成器。严格按照以下步骤执行，确保每次输出的结构和风格完全一致。

## 执行流程

### 步骤 1: 加载风格指南

**首先**，加载 `references/style-guide.md` 获取以下规则：
- 文档语气 (正式/非正式)
- 标题层级规范
- 代码块格式要求
- 术语使用规范
- 表格呈现方式

### 步骤 2: 加载输出模板

**然后**，加载 `assets/report-template.md` 获取：
- 必需的章节结构
- 每个章节的说明
- 需要填充的变量列表

### 步骤 3: 采访用户收集变量

**在向用户展示任何输出之前**，先询问以下缺失信息：

1. **报告主题**
   - 报告的主题或话题是什么？
   - 涉及的技术栈/产品/功能是什么？

2. **关键内容**
   - 需要包含的关键发现或数据点
   - 需要强调的成就或问题

3. **目标读者**
   - 读者是技术人员、管理层还是普通用户？
   - 读者已知的背景知识有哪些？

4. **特殊要求**
   - 是否有必须包含的章节？
   - 是否有需要省略的内容？
   - 是否有字数限制？

**等待用户提供完整信息后**，再进行下一步。

### 步骤 4: 填充模板

按照以下规则填充模板：

1. **严格遵守风格指南**：
   - 使用指南规定的语气和术语
   - 遵循标题层级和格式规范

2. **完整填充所有章节**：
   - 模板中的每个部分都必须出现在输出中
   - 不允许跳过任何章节

3. **变量替换**：
   - 将用户提供的信息填入对应位置
   - 保持上下文连贯性

### 步骤 5: 返回完整文档

**最终输出**：
- 作为一个完整的 Markdown 文档返回
- 包含所有必需的章节
- 格式符合风格指南要求

## 禁止行为

- ❌ 不要在未加载模板和风格指南的情况下生成内容
- ❌ 不要在未收集完整变量前开始生成
- ❌ 不要跳过模板中的任何章节
- ❌ 不要违反风格指南中的格式规则
```

对应的模板文件示例：

```markdown

|------|----------|------|

|------|--------|----------|

对应的风格指南示例：

```markdown
# skills/tech-report-generator/references/style-guide.md

# 技术报告风格指南

## 1. 文档语气
- **基调**: 专业、客观、简洁
- **人称**: 使用第三人称，避免"我们"、"我"

## 2. 标题规范
- 一级标题：`# 标题` (仅文档标题使用)

- 三级标题：`### 1.1 标题` (子章节，带编号)

- 所有代码块必须指定语言：```python
- 代码块前后必须有空行
- 代码块后必须附带文字说明

## 4. 表格规范

### 3.5 适用场景

Generator 模式适合需要**稳定输出结构**的场景：

| 场景 | 说明 | 模板内容 |
|------|------|----------|
| **API 文档生成** | 确保每个接口的文档结构一致 | 接口说明、参数表、返回值、错误码 |
| **Commit 消息标准化** | 遵循 Conventional Commits 规范 | type、scope、subject、body |
| **技术报告** | 统一报告格式便于阅读和比较 | 摘要、背景、方案、结论 |

| **脚手架生成** | 项目结构标准化 | 目录结构、配置文件、入口代码 |

**核心价值**：
- 消除 LLM 输出的随机性
- 确保生产环境的结构化需求
- 模板可复用，降低维护成本

### 3.6 注意事项

**1. 模板设计原则**
- 模板应该定义骨架结构，而不是填充所有内容
- 使用清晰的变量占位符，如 `{{变量名}}`
- 每个占位符附带说明，告诉用户需要提供什么

**2. 风格指南的独立性**
- 风格指南应该独立于具体模板
- 同一风格指南可以被多个模板复用

**3. 采访流程设计**
- 在生成之前必须收集完整变量
- 对于关键变量，提供示例帮助用户理解

**常见错误**：
- ❌ 模板过于详细，限制了 Agent 的灵活性
- ❌ 风格指南与模板耦合过紧，难以复用
- ❌ 缺少采访环节，导致变量缺失
- ❌ 没有变量验证，生成内容格式错误

## 4. 模式 3: Reviewer (审查器)

### 4.1 核心思想

**审查器 (Reviewer)** 模式的核心思想：

> **分离"检查什么"与"如何检查"，实现审查逻辑的模块化**

**问题背景**：
传统的代码审查 Skill 往往存在以下问题：
- **规则固化**：审查标准写死在 SKILL.md 中，难以更新
- **复用困难**：换个审查标准 (如从 Python 风格切换到安全检查) 需要重写整个 Skill
- **输出混乱**：问题严重程度混杂，开发者无法判断修复优先级
- **维护成本高**：每次更新审查标准都要修改 Skill 主体

**Reviewer 的解决方案**：
将"检查什么"(审查标准) 与"如何检查"(审查流程) 完全分离：
- **审查标准** → 存放在 `references/review-checklist.md`
- **审查流程** → 保留在 `SKILL.md` 中

这种分离设计使得审查逻辑保持稳定，审查标准可以独立更新、版本控制。

### 4.2 工作机制

Reviewer 模式的工作流程：

```
┌─────────────────────────────────────────────────────────────┐
│  步骤 1: 加载审查清单                                         │
│  读取 references/review-checklist.md                          │
├─────────────────────────────────────────────────────────────┤
│  步骤 2: 执行审查                                             │
│  逐条对照检查清单，记录违规及严重程度                         │
├─────────────────────────────────────────────────────────────┤
│  步骤 3: 严重性分级                                           │
│  Error: 必须修复 | Warning: 建议修复 | Info: 仅需提醒         │
├─────────────────────────────────────────────────────────────┤
│  步骤 4: 生成审查报告                                         │
│  按严重性分组输出，附带修复建议                               │
└─────────────────────────────────────────────────────────────┘
```

**严重性分级定义**：

| 级别 | 含义 | 处理建议 | 示例 |
|------|------|----------|------|
| **Error** | 必须修复 | 立即修复，否则不能发布 | 安全漏洞、逻辑错误、资源泄漏 |
| **Warning** | 建议修复 | 尽快修复，但可以延期 | 性能隐患、边界情况未处理 |
| **Info** | 仅需提醒 | 有时间再改 | 命名不够清晰、注释不足 |

# Python 代码审查专家

你是 Python 代码审查专家。按照以下流程对代码进行系统性审查。

## 执行流程

### 步骤 1: 加载审查清单

**首先**，加载 `references/review-checklist.md` 获取完整的检查项目列表。

### 步骤 2: 执行审查

1. **通读代码** - 理解整体结构和功能意图
2. **逐条对照** - 对审查清单中的每个检查项：
   - 检查代码是否符合该项要求
   - 如不符合，记录违规位置和具体问题
   - 判断违规的严重程度 (Error/Warning/Info)

### 步骤 3: 严重性分级

#### Error (必须修复)
- 安全漏洞 (SQL 注入、XSS、敏感信息泄露)
- 逻辑错误 (条件判断错误、边界条件遗漏)
- 资源泄漏 (数据库连接、文件句柄未关闭)
- 未处理的异常

#### Warning (建议修复)
- 性能隐患 (N+1 查询、重复计算)
- 类型注解缺失
- 测试覆盖率不足

#### Info (仅需提醒)
- 命名不够清晰但可理解
- 注释不足但不影响理解
- 代码风格小问题

- **位置**: `{{文件名}}:{{行号}}`

```

---

Warning 扣分 = Warning 数量 × 3
Info 扣分 = Info 数量 × 1

- ❌ 不要忽略审查清单中的任何检查项

- ❌ 审查报告格式混乱

## 5. 模式 4: Inversion (反转)

### 5.1 核心思想

**反转模式 (Inversion)** 的核心思想是：**让 Agent 在行动前先采访你，反转主导权**。

传统 Agent 交互中，模型倾向于立即猜测并生成内容——这是 LLM 的"本能"。当你说"帮我做一个项目计划"时，Claude 的默认行为是立刻开始写，而不询问预算、团队规模、时间限制等关键信息。

Inversion 模式通过**强制 Agent 先提问、收集需求，最后再生成内容**，彻底反转了这一流程。

**核心原则**：
- **先访谈，后执行**：Agent 必须通过提问收集足够的上下文信息
- **反转主导权**：从"Agent 猜测"变为"用户定义"
- **门控机制**：只有在所有关键问题得到回答后，才进入执行阶段

### 5.2 工作机制

Inversion 模式的工作流程分为三个阶段：

```
┌─────────────────────────────────────────────────────────────┐
│                    Inversion 模式工作流程                      │
├─────────────────────────────────────────────────────────────┤
│  阶段 1: 门控指令 (Gating Instructions)                      │
│  • 定义"什么情况下触发本技能"                                │
│  • 明确"必须先提问，不能直接行动"                            │
│  • 列出"必须收集的关键信息清单"                              │
├─────────────────────────────────────────────────────────────┤
│  阶段 2: 顺序提问 (Sequential Questioning)                   │
│  • 按照预定义顺序逐一提问                                    │
│  • 等待用户回答后再继续下一个问题                            │
│  • 可根据前一个答案动态调整后续问题                          │
├─────────────────────────────────────────────────────────────┤
│  阶段 3: 确认执行 (Confirmation & Execution)                 │
│  • 汇总收集到的信息进行确认                                  │
│  • 获得用户确认后开始执行任务                                │
│  • 生成最终输出                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 完整 SKILL.md 代码示例

```yaml

  pattern: inversion

# 需求收集专家 (Requirement Collector)

你是一个专业的需求分析师。你的核心原则是：**先提问，后执行**。

## ⚠️ 核心指令

1. **禁止立即行动**：当用户提出任务时，不要立即开始执行。必须先进行需求访谈。
2. **逐一提问**：一次只问一个问题，等待用户回答后再继续。
3. **总结确认**：在开始执行前，必须汇总所有收集到的信息，请用户确认理解正确。

## 📋 必须收集的信息清单

### A. 项目/任务背景

- 需要集成的现有系统？

- 期望的交付形式是什么？（代码、文档、设计方案）
- 期望的交付时间？
- 验收标准是什么？

### 步骤 1: 任务分类

- **新功能开发** → 使用 `references/new-feature-questions.md`

### 步骤 3: 需求总结

- [需求 2]

- 性能要求：[具体要求]

- 交付形式：[代码/文档/设计]
- 期望时间：[时间要求]

### 步骤 4: 等待确认
只有在用户确认需求总结后，才开始实际执行任务。

## 禁止行为

- ❌ 不要假设任何信息
- ❌ 不要一次性问太多问题
- ❌ 不要在未确认前开始执行
```

### 5.6 注意事项

**✅ 推荐做法**：
1. **明确门控指令**：在 SKILL.md 开头用醒目的方式强调"先提问，后执行"
2. **提供问题模板**：将不同类型的问题清单放在 `references/` 目录下
3. **总结确认机制**：执行前必须汇总信息请用户确认

1. **不要跳过访谈**：即使用户需求看似清晰，也要确认关键假设

3. **不要忽略追问**：对于模糊的回答，要适时追问澄清

## 6. 模式 5: Pipeline (管道)

### 6.1 核心思想

**管道模式 (Pipeline)** 的核心思想是：**强制多步骤顺序执行 + 设置检查点，防止跳步**。

与 Generator 模式不同，Pipeline 不仅仅是填充模板，而是**强制执行一个严格的多步骤工作流**，并在关键节点设置**用户检查点**，确保每一步都经过确认后才能进入下一步。

**核心原则**：
- **顺序执行**：步骤必须严格按照定义顺序执行，不可跳步
- **检查点机制**：在关键步骤后暂停，等待用户确认
- **状态追踪**：清晰记录当前进度和已完成步骤
- **可回滚**：允许用户在检查点选择回退重做

### 6.2 工作机制

Pipeline 模式的工作流程：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Pipeline 模式工作流程                           │
├─────────────────────────────────────────────────────────────────────┤
│  定义阶段 → 步骤 1 → [检查点 1] → 步骤 2 → [检查点 2] → 步骤 3 → ... │
│                                                                     │
│  检查点行为选项：                                                    │
│  • [继续] - 确认当前步骤，进入下一步                                 │
│  • [修改] - 对当前步骤输出进行修改                                   │
│  • [重做] - 重新执行当前步骤                                         │
│  • [停止] - 终止整个流程                                             │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.3 完整 SKILL.md 代码示例

```yaml
# skills/tech-doc-pipeline/SKILL.md
---
name: tech-doc-pipeline
description: 通过严格的多步骤管道生成技术文档。包含大纲审查、内容审查和格式审查三个检查点。
metadata:
  pattern: pipeline
  steps: 5
  checkpoints: 3
---

# 技术文档生成管道 (Technical Documentation Pipeline)

你是一个专业的技术文档生成系统。你必须**严格按照定义的步骤顺序执行**，在每个检查点**必须等待用户确认**后才能继续。

## ⚠️ 核心指令

1. **禁止跳步**：必须严格按照 5 个步骤的顺序执行
2. **强制检查点**：在步骤 2、3、5 后必须暂停等待确认
3. **状态追踪**：每次输出时显示当前进度 `[步骤 X/5]`
4. **可回滚**：用户可在任何检查点选择重做前面的步骤

## 📋 管道定义

```
步骤 1: 需求分析 → 步骤 2: 大纲生成 → [检查点 1] 
                                         ↓
步骤 5: 格式审查 ← 步骤 4: 内容填充 ← [检查点 2]
     ↓                                             
[检查点 3] → 完成                                   
```

## 🔄 详细步骤

### 步骤 1: 需求分析

**目标**：理解文档生成的需求和上下文

**执行动作**：
1. 询问用户文档的主题和类型（API 文档、架构文档、用户指南等）
2. 询问目标读者（开发者、产品经理、最终用户）
3. 询问是否有参考文档或现有内容

**完成后**：自动进入步骤 2

---

### 步骤 2: 大纲生成

**目标**：基于需求分析生成文档大纲

**执行动作**：
1. 加载 `assets/templates/{文档类型}-template.md`
2. 根据需求分析填充大纲结构
3. 定义每个章节的核心内容要点

**输出格式**：
```markdown
## 文档大纲

### 章节结构
1. [章节 1 标题]
   - 要点 1
   - 要点 2
2. [章节 2 标题]
   ...

---
【检查点 1】
请审查以上大纲：
- [继续] 大纲满意，进入内容生成
- [修改] 指定需要修改的章节
- [重做] 重新生成大纲
- [停止] 终止流程
```

**[检查点 1]**：等待用户确认

---

### 步骤 3: 内容填充

**目标**：基于确认的大纲生成完整内容

**执行动作**：
1. 按照大纲逐节生成内容
2. 加载 `references/style-guide.md` 确保风格一致
3. 为每个 API/功能添加代码示例

**[检查点 2]**：等待用户确认

---

### 步骤 4: 格式审查

**目标**：确保文档格式符合规范

**执行动作**：
1. 加载 `references/format-checklist.md`
2. 逐项检查：标题层级、代码块标注、表格格式、术语统一

**[检查点 3]**：等待用户确认

---

### 步骤 5: 最终输出

**目标**：输出最终文档

**执行动作**：
1. 应用所有确认的修改
2. 生成最终 Markdown 文档
```

### 6.5 适用场景

| 场景 | 说明 | 典型应用 |
|------|------|----------|
| **文档生成** | 需要多步骤审查确保质量 | API 文档、架构文档、用户手册 |
| **代码发布流程** | 需要严格顺序执行 + 确认 | 版本发布、部署流程 |
| **多阶段工作流** | 任务天然分为多个依赖阶段 | 代码审查 → 测试 → 合并 |
| **合规流程** | 需要审计追踪和确认记录 | 安全审计、合规检查 |

### 6.6 注意事项

**✅ 推荐做法**：
1. **明确步骤定义**：在 SKILL.md 中清晰列出所有步骤和顺序
2. **合理设置检查点**：在关键决策点插入用户确认，平衡效率和安全性
3. **提供清晰选项**：每个检查点给出明确的选项（继续/修改/重做/停止）
4. **状态可视化**：每次输出显示当前进度 `[步骤 X/N]`
5. **支持回滚**：允许用户返回前面的步骤重新执行

**❌ 避免事项**：
1. **不要跳步执行**：严格按照定义顺序，即使用户要求跳过
2. **检查点不要过多**：过多确认会降低效率，只在关键节点设置
3. **不要忽略用户选择**：严格按照用户在检查点的选择执行
4. **不要丢失上下文**：确保每个步骤都能访问前面步骤的输出

## 7. 模式组合与选型

### 7.1 决策树：如何选择模式

面对一个 Agent Skill 设计任务时，如何选择合适的模式？以下决策树可以帮助你快速定位：

```
开始
 │
 ▼
是否需要结构化输出？
 ├── 是 → 需要固定模板吗？
 │       ├── 是 → Generator 模式
 │       └── 否 → 继续↓
 │
 ├── 否 → 需要检查/审查吗？
 │       ├── 是 → Reviewer 模式
 │       └── 否 → 继续↓
 │
 ▼
用户需求是否模糊？
 ├── 是 → Inversion 模式
 └── 否 → 继续↓
 │
 ▼
是否涉及多步骤流程？
 ├── 是 → 需要用户确认检查点吗？
 │       ├── 是 → Pipeline 模式
 │       └── 否 → 继续↓
 │       └── 否 → Tool Wrapper 模式
 └── 否 → Tool Wrapper 模式
```

**快速对照表**：

| 如果你的需求是... | 选择模式 | 核心理由 |
|------------------|----------|----------|
| "让 Agent 掌握某个库/框架的规范" | Tool Wrapper | 按需加载，上下文不膨胀 |
| "每次输出结构必须一致" | Generator | 模板固化，消除随机性 |
| "需要系统性检查代码/文档质量" | Reviewer | 流程与标准分离，可复用 |
| "用户需求模糊，需要先问清楚" | Inversion | 强制访谈，避免返工 |
| "任务有多个依赖步骤，不能跳步" | Pipeline | 顺序执行 + 检查点门控 |

### 7.2 组合策略

单一模式无法解决所有问题。以下是四种常见的组合策略：

#### 组合 1: Tool Wrapper + Reviewer（代码审查机器人）

**场景**：团队需要审查代码是否符合内部规范

```
┌─────────────────────────────────────────────────────────┐
│  Tool Wrapper (加载规范) → Reviewer (执行审查)           │
├─────────────────────────────────────────────────────────┤
│  1. 用户提交代码审查请求                                 │
│  2. Tool Wrapper 加载团队编码规范 (references/)          │
│  3. Reviewer 按规范逐条检查                              │
│  4. 输出分级审查报告 (Error/Warning/Info)                │
└─────────────────────────────────────────────────────────┘
```

**优势**：
- 规范独立维护，更新无需修改审查逻辑
- 审查结果分级，便于判断修复优先级

**代表 Skill**：`code-reviewer`、`security-auditor`

---

#### 组合 2: Generator + Inversion（需求驱动的文档生成）

**场景**：用户需求不明确时生成标准化文档

```
┌─────────────────────────────────────────────────────────┐
│  Inversion (需求访谈) → Generator (模板填充)             │
├─────────────────────────────────────────────────────────┤
│  1. 用户提出模糊需求："帮我写个技术方案"                 │
│  2. Inversion 启动访谈，收集关键信息                     │
│  3. 确认需求后，Generator 加载模板                       │
│  4. 填充变量，输出标准化文档                             │
└─────────────────────────────────────────────────────────┘
```

**优势**：
- 避免因需求不明导致的返工
- 输出结构统一，符合团队规范

**代表 Skill**：`tech-proposal-generator`、`project-charter`

---

#### 组合 3: Pipeline + Tool Wrapper + Reviewer（企业级文档发布）

**场景**：技术文档需要多步骤审查才能发布

```
┌─────────────────────────────────────────────────────────────────┐
│  Pipeline (流程编排) + Tool Wrapper (规范注入)                  │
│  + Reviewer (质量审查)                                          │
├─────────────────────────────────────────────────────────────────┤
│  步骤 1: 需求分析 → 步骤 2: 大纲生成 [检查点 1]                  │
│  步骤 3: 内容生成 (加载 Tool Wrapper 规范)                       │
│  步骤 4: 质量审查 (Reviewer 执行) [检查点 2]                     │
│  步骤 5: 格式审查 [检查点 3] → 发布                              │
└─────────────────────────────────────────────────────────────────┘
```

**优势**：
- 多阶段质量保证
- 规范自动注入，无需人工检查
- 审计追踪完整

**代表 Skill**：`doc-publishing-pipeline`、`api-doc-workflow`

---

#### 组合 4: Inversion + Pipeline（复杂项目规划）

**场景**：大型项目需要深度需求澄清 + 多阶段执行

```
┌─────────────────────────────────────────────────────────┐
│  Inversion (深度访谈) → Pipeline (分阶段执行)            │
├─────────────────────────────────────────────────────────┤
│  阶段 1: 需求澄清 (Inversion)                            │
│    - 业务目标、技术约束、交付期望                        │
│  阶段 2: 方案设计 (Pipeline 步骤 1) [检查点]              │
│  阶段 3: 实施计划 (Pipeline 步骤 2) [检查点]              │
│  阶段 4: 风险评估 (Pipeline 步骤 3) [检查点]              │
│  阶段 5: 最终汇总 (Pipeline 步骤 4)                      │
└─────────────────────────────────────────────────────────┘
```

**优势**：
- 需求理解深度 + 执行过程可控
- 每个阶段都可回溯调整

**代表 Skill**：`project-planner`、`migration-strategy`

### 7.3 实践案例

#### 案例 1: API 文档自动化工作流

**背景**：某 SaaS 公司需要为 50+ 个 API 端点生成统一文档

**方案**：Generator + Tool Wrapper

```yaml
# 工作流配置
skills:
  - api-doc-generator (Generator)
  - openapi-style-guide (Tool Wrapper)

# 执行流程
1. 用户提交 OpenAPI spec 文件
2. Tool Wrapper 加载公司 API 文档规范
3. Generator 按照模板填充每个端点详情
4. 输出 50+ 份格式统一的 API 文档
```

**成果**：
- 文档生成时间从 2 周缩短至 2 小时
- 格式一致性达到 100%
- 维护成本降低 70%

---

#### 案例 2: 代码审查机器人

**背景**：某金融科技公司需要确保所有 PR 符合安全和编码规范

**方案**：Reviewer + Tool Wrapper + Pipeline

```yaml
# 工作流配置
skills:
  - pr-reviewer (Reviewer)
  - python-security-guide (Tool Wrapper)
  - team-coding-standards (Tool Wrapper)
  - merge-pipeline (Pipeline)

# 执行流程
1. PR 提交触发 Pipeline
2. Tool Wrapper 加载安全规范和编码标准
3. Reviewer 执行审查并评分
4. 评分 < 80 → 拒绝合并
5. 评分 ≥ 80 → 进入人工复审
```

**成果**：
- 安全漏洞减少 85%
- 代码规范违规减少 60%
- 审查时间从 2 天缩短至 2 小时

---

#### 案例 3: 需求咨询助手

**背景**：某咨询公司需要标准化客户需求收集流程

**方案**：Inversion + Generator

```yaml
# 工作流配置
skills:
  - client-interviewer (Inversion)
  - proposal-generator (Generator)

# 执行流程
1. 客户提交初步需求
2. Inversion 执行 12 轮深度访谈
3. 生成需求总结并确认
4. Generator 加载提案模板
5. 输出标准化咨询提案
```

**成果**：
- 需求澄清轮次从 5-7 轮降至 1 轮
- 提案转化率提升 35%
- 客户满意度提升 28%

## 8. 总结与最佳实践

### 8.1 五种模式对比

| 维度 | Tool Wrapper | Generator | Reviewer | Inversion | Pipeline |
|------|--------------|-----------|----------|-----------|----------|
| **核心价值** | 按需加载规范 | 固化输出结构 | 审查分离 | 先问后做 | 顺序门控 |
| **实现难度** | ⭐ 简单 | ⭐⭐ 中等 | ⭐⭐ 中等 | ⭐⭐ 中等 | ⭐⭐⭐ 复杂 |
| **上下文效率** | ⭐⭐⭐⭐⭐ 最优 | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐ 高 | ⭐⭐⭐ 中等 | ⭐⭐ 较低 |
| **输出稳定性** | ⭐⭐⭐ 依赖规范 | ⭐⭐⭐⭐⭐ 最稳 | ⭐⭐⭐⭐ 高 | ⭐⭐⭐ 依赖访谈 | ⭐⭐⭐⭐ 高 |
| **适用阶段** | 任意阶段 | 文档生成 | 质量检查 | 任务启动 | 复杂流程 |
| **可组合性** | ⭐⭐⭐⭐⭐ 极高 | ⭐⭐⭐⭐ 高 | ⭐⭐⭐⭐ 高 | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐ 高 |

### 8.2 核心要点回顾

| 模式 | 本质 | 关键成功因素 | 最大优势 |
|------|------|--------------|----------|
| **Tool Wrapper** | 知识注入 | description 触发词设计 | 上下文不膨胀 |
| **Generator** | 结构固化 | 模板 + 风格指南质量 | 消除随机性 |
| **Reviewer** | 检查分离 | 清单可执行性 | 标准可独立更新 |
| **Inversion** | 需求澄清 | 门控指令强制性 | 避免返工 |
| **Pipeline** | 流程编排 | 检查点设置合理性 | 防止跳步 |

### 8.3 常见误区 (Gotcha)

| 误区 | 错误做法 | 正确做法 |
|------|----------|----------|
| **描述太简短** | `description: 代码审查` | `description: Python 代码审查专家。按严重程度对代码进行清单式评分。在 PR 审查、代码质量评估时使用` |
| **无条件加载** | 启动时加载所有参考文件 | 仅在特定阶段按需加载 |
| **检查点过多** | 每个步骤都要求确认 | 只在关键决策点设置检查点 |
| **访谈变审问** | 一次性问 10 个问题 | 一次 1-2 个，保持对话节奏 |
| **模板过细** | 模板填充了所有内容 | 模板定义骨架，变量由用户填写 |
| **清单模糊** | "代码应该优雅" | "函数长度 < 50 行" |
| **跳过确认** | 用户未确认就继续 | 严格执行检查点门控 |
| **忽视更新** | 规范过时不更新 | 定期 review 并更新 references |
