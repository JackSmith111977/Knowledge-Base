## 1. SKILL 概述

### 1.1 什么是 SKILL

**SKILL（技能）** 是 AI Agent 与外部世界交互的**能力模块**，是将专业知识、工作流程和最佳实践转化为可复用能力的标准化格式。

**核心价值：**

> Teach AI your way of working（教会 AI 按你的方式工作）

**通俗理解：**

```
想象你请了一位私人助理（Agent）：
- 这位助理很聪明，能听懂你的话、分析问题、制定计划
- 但你不会指望他凭空帮你订机票——他需要调用订票系统
- 你也不会指望他直接帮你写代码——他需要打开 IDE

这位助理是 Agent，而订票系统、IDE 这些工具的使用说明，就是 Skill。

Skill 的本质：岗位培训大礼包
- PDF + 流程图 + SOP + 话术模板 + 常用脚本
- 告诉他："当老板让你做这类事情时，就按这个文件夹里的方法来做"
```

### 1.2 为什么需要 SKILL

**LLM（大语言模型）的能力边界：**

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM 核心限制                              │
├─────────────────────────────────────────────────────────────┤
│  1. 信息滞后：知识截止于训练数据，无法获取实时信息           │
│  2. 缺乏执行能力：只能输出文字，无法操作外部系统             │
│  3. 静态知识：内部知识难以动态更新，需要重新训练             │
│  4. 缺乏真正逻辑推理：基于统计规律，非真正逻辑推理           │
└─────────────────────────────────────────────────────────────┘
```

**Skill 解决的问题：**

| 问题 | 传统方式 | Skill 方式 |
|------|----------|------------|
| **知识传递** | 每次重复描述需求 | 一次定义，永久应用 |
| **输出一致性** | 依赖 AI 临场发挥 | 标准化流程，专家级输出 |
| **复杂任务** | 多轮对话逐步指导 | 堆叠 Skills 自动处理 |
| **组织知识沉淀** | 文档散落各处 | 打包为可执行 Skill |

### 1.3 Skill 与其他方案对比

| 方案 | 定义 | 优势 | 局限 |
|------|------|------|------|
| **Prompt** | 一次性指令 | 简单直接 | 每次都要重新教，无法复用 |
| **MCP/Tools** | 外部工具/API 接口 | 能调用真实系统 | 只定义"能做什么"，不定义"怎么用" |
| **Skill** | 标准操作流程（SOP） | 包含完整工作流程和最佳实践 | 需要预先定义场景 |

**关系图：**

```
┌─────────────────────────────────────────────────────────────┐
│                      AI Agent 架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐                                           │
│   │    LLM      │ ← 大脑：理解、推理、规划                   │
│   │  (大脑)     │                                           │
│   └──────┬──────┘                                           │
│          │                                                  │
│          ├──→ ┌─────────────┐                               │
│          │    │   Skills    │ ← 岗位培训手册：怎么做         │
│          │    │  (SOP 流程)  │                               │
│          │    └─────────────┘                               │
│          │                                                  │
│          └──→ ┌─────────────┐                               │
│               │  MCP/Tools  │ ← 工具接口：能做什么           │
│               │  (API/命令)  │                               │
│               └─────────────┘                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 Skill 的核心价值

**四大核心价值：**

| 价值 | 说明 | 实际收益 |
|------|------|----------|
| **Expert output, every time** | 专家级输出，每次都一致 | 一次定义，永久应用，确保输出稳定可靠 |
| **Capture what your organization knows** | 捕获组织知识 | 将公司流程、最佳实践打包，团队工作一致 |
| **Build once, use everywhere** | 一次构建，处处运行 | 在 Claude.ai、Claude Code、API 多平台无缝运行 |
| **Stack skills for complex work** | 堆叠技能处理复杂任务 | 组合多个 Skill 处理多步骤工作流 |

## 2. 核心概念辨析

### 2.1 LLM、Agent、Skill、MCP 关系

```
┌─────────────────────────────────────────────────────────────┐
│                    概念层次结构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: LLM（大语言模型）                                  │
│  ────────────────────────────                                │
│  核心能力：语言理解、文本生成、推理分析                      │
│  限制：只能输出文字，无法直接操作外部世界                    │
│  代表：GPT-4、Claude、Gemini、Qwen                          │
│                                                             │
│         + LLM 配备能力                                        │
│         ↓                                                   │
│                                                             │
│  Layer 2: Agent（智能体）                                    │
│  ────────────────────────────                                │
│  公式：Agent = LLM + 记忆 + 规划 + 工具 + 自我反思            │
│  核心能力：自主感知环境、制定计划、调用工具、执行任务        │
│  代表：Claude Code、AutoGen、LangChain Agent                │
│                                                             │
│         + Agent 能力扩展                                      │
│         ↓                                                   │
│                                                             │
│  Layer 3: Skill（技能）                                      │
│  ────────────────────────────                                │
│  定义：Agent 与外部世界交互的能力单元                         │
│  组成：工具定义 + 执行逻辑 + 使用说明                         │
│  示例：天气查询、邮件发送、代码执行、安全扫描                │
│                                                             │
│         + Skill 调用外部工具                                  │
│         ↓                                                   │
│                                                             │
│  Layer 4: MCP/Tools（工具协议）                              │
│  ────────────────────────────                                │
│  定义：Agent 调用外部 API/系统的标准化接口                    │
│  示例：文件系统、数据库、HTTP API、Git 操作                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Agent 公式详解

```
Agent = LLM（大脑） + 记忆 + 工具/Skills + 规划

┌─────────────────────────────────────────────────────────────┐
│ Agent 四大核心能力                                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 环境感知                                                  │
│     - 通过视觉、语音、API 等多模态接口获取环境数据             │
│     - 示例：读取文件、查询数据库、调用天气 API                │
│                                                             │
│  2. 智能决策                                                  │
│     - 运用深度学习模型和强化学习算法进行复杂决策             │
│     - 示例：任务拆解、优先级判断、风险评估                  │
│                                                             │
│  3. 任务执行                                                  │
│     - 调用 API 工具库或操控物理设备完成实际工作               │
│     - 示例：发送邮件、执行代码、生成报告                    │
│                                                             │
│  4. 持续进化                                                  │
│     - 具备在线学习和迁移学习能力，实现性能持续提升           │
│     - 示例：从历史任务中学习、优化执行策略                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 Agent 的四种形态

| 形态 | 说明 | 代表技术 |
|------|------|----------|
| **Reflection（反思模式）** | 通过模型自身反思来改进任务执行 | ReAct 框架、Self-Refine 算法 |
| **Tool use（工具调用）** | 调用外部工具或库来解决任务 | Function Calling、API 集成 |
| **Planning（规划模式）** | 提前计划和组织步骤来提升效率 | 思维链（CoT）、思维树（ToT） |
| **Multi-agent collaboration（多智能体协作）** | 多个智能体协作完成任务 | A2A 协议 |

## 3. 架构设计

### 3.1 Skill 三层加载架构（渐进式披露）

**核心设计理念：** 按需加载，节省 Token

```
┌─────────────────────────────────────────────────────────────┐
│                   Skill 三层架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: Metadata（元数据层）                               │
│  ─────────────────────────────                               │
│  内容：name + description                                    │
│  加载时机：始终在上下文中                                     │
│  Token 配额：约 100 词                                          │
│  作用：决定技能何时触发                                      │
│                                                             │
│  ↓ 触发后加载                                                │
│                                                             │
│  Layer 2: SKILL.md body（技能主体）                          │
│  ─────────────────────────────                               │
│  内容：具体指令、使用指南、注意事项                           │
│  加载时机：技能触发后                                        │
│  Token 配额：< 5000 词（建议< 500 行）                           │
│  作用：核心工作流程                                          │
│                                                             │
│  ↓ 按需加载                                                  │
│                                                             │
│  Layer 3: Bundled resources（捆绑资源）                      │
│  ─────────────────────────────                               │
│  内容：scripts/、references/、assets/                        │
│  加载时机：按需加载                                          │
│  Token 配额：无限（scripts 可执行而不读入上下文）              │
│  作用：详细参考和可执行代码                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**架构优势：**

```
传统 Prompt 方式：
┌─────────────────────────────────────────┐
│ 一次性加载所有指令（2000+ Token）         │
│ - 大部分内容与当前任务无关               │
│ - Token 浪费严重                         │
│ - 上下文窗口占用大                       │
└─────────────────────────────────────────┘

Skill 渐进式披露：
┌─────────────────────────────────────────┐
│ 仅加载 Metadata（100 Token）             │ ← 平时
│        ↓ 触发后                          │
│ 加载 SKILL.md body（2000 Token）         │ ← 需要时
│        ↓ 按需                            │
│ 加载 resources（不限）                   │ ← 必须时
└─────────────────────────────────────────┘
```

### 3.2 Skill 标准目录结构

```
skill-name/
├── SKILL.md              # 主文件（必需）
├── examples/             # 示例目录（可选）
│   ├── example1.md
│   └── example2.md
├── references/           # 参考资料（可选）
│   ├── categories/       # 分类参考
│   ├── compliance/       # 合规规范
│   └── scripts/          # 脚本模板
├── utils/                # 工具文档（可选）
│   └── helpers.md
└── assets/               # 静态资源（可选）
    └── images/
```

| 平台 | 是否免费 | 默认路径 | 推荐人群 |

## 4. Skill 编写规范

### 4.1 SKILL.md 文件格式

**基本结构：**

```markdown
---
name: skill-name
description: 简短描述技能用途
aliases: [别名 1, 别名 2]  # 可选
triggers: [触发词 1, 触发词 2]  # 可选
---

# Skill 名称

## 用途

说明这个技能的适用场景，什么时候应该使用。

## 工作流程

### 步骤 1：步骤名称

具体操作说明，包括：
- 使用什么工具
- 如何执行
- 预期结果

### 步骤 2：步骤名称

...

## 输出格式

定义输出应该遵循的格式规范。

## 注意事项

- 需要特别注意的边界情况
- 常见错误及避免方法
- 安全和合规要求

## 示例

提供使用示例，帮助 AI 理解如何执行。
```

### 4.2 YAML Frontmatter 详解

**必需字段：**

```yaml
---
name: skill-name              # 技能唯一标识
description: 技能描述          # 显示在技能菜单中
---
```

**可选字段：**

```yaml
---
name: code-review 代码审查     # 英文 + 中文，便于搜索
description: 结构化代码审查指南
aliases: [review, code-review-assistant]  # 别名
triggers: [审查代码，review code, 检查代码]  # 触发词
version: 1.0.0                # 版本号
author: your-name             # 作者
---
```

**命名建议：**

```yaml
# 推荐：英文 + 中文（保留英文方便搜索）
name: remotion-best-practices 视频剪辑工具
name: web-design-guidelines Web 设计指南
name: seo-audit SEO 审计

# 不推荐：纯中文（搜索不便）
name: 视频剪辑工具

# 不推荐：纯英文（记忆困难）
name: remotion-best-practices
```

### 4.3 编写最佳实践

**✅ 推荐做法：**

```markdown
## 1. 明确触发条件

当用户请求以下类型的任务时，自动激活此 Skill：
- 代码审查
- API 文档生成
- 数据库迁移

## 2. 提供具体检查清单

### 代码审查清单

- [ ] 是否存在 SQL 注入风险
- [ ] 是否有适当的错误处理
- [ ] 是否遵循命名规范
- [ ] 是否添加了必要的注释

## 3. 定义输出格式

输出应包含：
1. 问题汇总（按严重程度排序）
2. 具体问题（文件：行号）
3. 修复建议（附代码示例）
```

**❌ 避免做法：**

```markdown
## 不好的示例

- 描述过于笼统："审查代码质量"
- 没有具体步骤："根据需要进行检查"
- 缺少输出格式："给出审查结果"
- Token 浪费：包含大量与当前任务无关的内容
```

### 4.4 完整 Skill 示例

**示例 1：代码解释器 Skill**

```markdown
---
name: explain-code 代码解释
description: 逐行解释代码功能和原理
aliases: [explain, code-explain]
---

# 代码解释器

## 用途

当用户请求解释代码时，自动使用此 Skill。
适用于：学习他人代码、理解复杂逻辑、代码审查前准备。

## 工作流程

### 步骤 1：阅读代码

使用 Read 工具读取目标文件：
```
read path/to/file.py
```

### 步骤 2：分析结构

识别代码的：
- 入口点（main 函数/程序起点）
- 核心函数/类
- 数据流向
- 依赖关系

### 步骤 3：逐行解释

按以下格式输出：

```markdown
## 代码结构
- 入口：main() 函数
- 核心模块：数据处理、API 调用、结果输出

## 逐行解释

### 第 1-10 行：导入模块
导入必要的库：requests 用于 HTTP 请求，json 用于数据解析...

### 第 11-30 行：数据获取函数
get_data() 函数负责从 API 获取数据，包含重试逻辑...

### 第 31-50 行：主处理逻辑
...
```

## 注意事项

- 避免过度技术术语，根据用户水平调整解释深度
- 重点解释"为什么"而不是"是什么"
- 指出潜在的代码异味和改进建议
```

**示例 2：安全扫描 Skill**

```markdown
---
name: security-scan 安全扫描
description: 使用 Semgrep 进行代码安全扫描
aliases: [security, scan]
---

# 安全扫描

## 用途

当用户请求安全检查、漏洞扫描时，自动使用此 Skill。

## 工作流程

### 步骤 1：确认扫描范围

询问用户：
- 扫描整个项目还是特定文件？
- 关注特定类型漏洞（SQL 注入、XSS、认证问题）？

### 步骤 2：执行 Semgrep 扫描

```bash
# 全量扫描
semgrep --config auto .

# 针对特定语言
semgrep --config auto --python .

# 针对特定规则
semgrep --config p/sql-injection .
```

### 步骤 3：分析结果

按严重程度分类：
- 🔴 高危：需要立即修复
- 🟡 中危：建议修复
- 🟢 低危：知情即可

### 步骤 4：生成报告

输出格式：
```markdown
## 安全扫描结果

### 高危问题（2 个）
1. SQL 注入风险 (user_service.py:45)
   - 问题：直接使用字符串拼接 SQL
   - 修复：使用参数化查询

### 中危问题（5 个）
...
```

## 注意事项

- 扫描前确认已安装 Semgrep
- 结果需要人工复核，避免误报
- 敏感项目不要上传到云端扫描
```

## 5. 加载与触发机制

### 5.1 技能触发方式

**显式触发：**

```
用户输入：/skill-name 或使用别名

示例：
/explain-code main.py
/code-review src/
/security-scan
```

**隐式触发：**

```
用户请求包含触发词时自动激活

示例：
用户："帮我解释一下这个函数"
→ 自动触发 explain-code Skill

用户："检查这段代码有没有安全问题"
→ 自动触发 security-scan Skill
```

### 5.2 技能选择策略

**Agent 选择技能的决策过程：**

```
1. 解析用户意图
   ↓
2. 匹配 Metadata 中的 description 和 triggers
   ↓
3. 评估技能相关性
   ↓
4. 加载匹配技能的完整内容（SKILL.md body）
   ↓
5. 按需加载 resources（scripts、references）
   ↓
6. 按 Skill 定义的工作流程执行
```

### 5.3 技能堆叠（Stacking）

**复杂任务需要多个 Skill 协作：**

```
用户："分析这个项目，生成技术文档，然后审查代码质量"

触发的 Skills：
1. code-analyzer → 分析代码结构和依赖
2. doc-generator → 生成 API 文档
3. code-reviewer → 审查代码质量

执行流程：
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  code-      │ →  │  doc-       │ →  │  code-      │
│  analyzer   │    │  generator  │    │  reviewer   │
└─────────────┘    └─────────────┘    └─────────────┘
     ↓                    ↓                    ↓

### 6.3 数据库迁移助手

**SKILL.md：**

```markdown
---
name: db-migration 数据库迁移助手
description: 生成可回滚的数据库迁移脚本
triggers: [数据库迁移，migration, 表结构变更]
---

# 数据库迁移助手

## 核心原则

**必须可回滚！** 每个迁移脚本必须有对应的回滚脚本。

## 工作流程

### 步骤 1：分析变更需求

- 确定涉及的表和字段
- 评估数据量级
- 识别潜在风险

### 步骤 2：生成迁移脚本

```sql
-- UP migration
ALTER TABLE users ADD COLUMN email VARCHAR(255);
CREATE INDEX idx_users_email ON users(email);

-- DOWN migration (必须提供！)
DROP INDEX idx_users_email ON users;
ALTER TABLE users DROP COLUMN email;
```

### 步骤 3：验证清单

- [ ] UP 脚本可执行
- [ ] DOWN 脚本可执行
- [ ] 数据不会丢失
- [ ] 大表变更已评估性能影响
```

## 7. 安全与最佳实践

### 7.1 安全风险

**已知安全事件：**

| 事件 | 描述 | 影响 |
|------|------|------|
| **ClawHavoc（1184 个恶意 Skill）** | 伪装成正常工具，实际窃取数据、注入恶意指令 | 数据泄露 |
| **WhatsApp 消息触发漏洞** | 精心构造的消息触发读取 .env 和 creds.json | API 密钥泄露 |
| **Moltbook 数据库泄露** | Agent 社交网络数据库未保护 | 35000 邮箱 +150 万 API Token |
| **Google 日历零点击漏洞** | 恶意日历事件触发远程代码执行 | 影响 10000+ 用户 |

### 7.2 安全最佳实践

**创建 Skill 时的安全考虑：**

```markdown
## 安全检查清单

- [ ] 不请求不必要的权限
- [ ] 敏感操作需要用户确认
- [ ] 不读取 .env、creds.json 等敏感文件
- [ ] 外部 API 调用需要验证 SSL 证书
- [ ] 执行的脚本需要沙箱隔离
```

**使用第三方 Skill 的注意事项：**

```
1. 审查 Skill 源码
   - 检查是否有可疑的文件读取操作
   - 检查是否有未授权的网络请求

2. 限制权限范围
   - 使用最小权限原则
   - 敏感命令需要确认

3. 定期更新
   - 关注 Skill 作者的安全公告
   - 及时更新到安全版本
```

### 7.3 性能考虑

**Token 优化策略：**

| 策略 | 说明 | 节省比例 |
|------|------|----------|
| 渐进式披露 | 按需加载，不一次性加载所有 | 60-80% |
| 资源外部化 | scripts 可执行不读入 | 视情况 |
| 精简元数据 | description 控制在 100 词内 | 10-20% |
| 避免重复 | 复用已有 Skill，不重复创建 | - |

### 7.4 使用建议

**个人开发者：**

```
1. 从简单 Skill 开始（代码解释、PR 描述生成）
2. 逐步积累个人 Skill 库
3. 定期整理和更新
```

**团队/企业：**

```
1. 建立 Skill 审核流程
2. 统一 Skill 存放位置（Git 仓库）
3. 定期安全审计
4. 建立 Skill 市场和分享机制
```

## 8. Claude Code Skill 最佳实践

> 本章节基于 Anthropic 官方文档、工程师实战经验分享整理
> 来源：Anthropic Engineering Blog、官方 Skill 仓库、开发者社区实践

### 8.1 Skill 的本质理解

**核心定义：**

Skill 不是"Markdown 文件"，而是一个**文件夹结构**，可以包含：
- `SKILL.md`（必需）：核心指令文件
- `scripts/`（可选）：可执行脚本（Python、Bash 等）
- `references/`（可选）：按需加载的参考文档
- `assets/`（可选）：模板、字体、图标等资源

**关键思维转变：**

```
❌ 错误理解：Skill = 写一个 SKILL.md 文件
✅ 正确理解：Skill = 设计一个文件夹结构，Claude 可以探索和操纵其中的文件
```

### 8.2 9 种 Skill 类型分类

根据 Anthropic 内部使用的数百个 Skills 归纳：

| 类型 | 用途 | 典型示例 | 优先级 |
|------|------|----------|--------|
| **1. 库和 API 参考** | 教 Claude 正确使用内部库/CLI/SDK | `billing-lib`、`internal-platform-cli` | ⭐⭐⭐⭐⭐ |
| **2. 产品验证** | 描述如何测试代码是否正常工作 | `signup-flow-driver`、`checkout-verifier` | ⭐⭐⭐⭐⭐ |
| **3. 数据获取与分析** | 连接数据系统和监控系统 | `funnel-query`、`cohort-compare` | ⭐⭐⭐⭐ |
| **4. 代码生成** | 根据规格说明生成代码 | `frontend-from-design`、`api-from-spec` | ⭐⭐⭐⭐ |
| **5. 文档生成** | 生成标准化文档 | `pr-description`、`api-doc-generator` | ⭐⭐⭐⭐ |
| **6. 代码审查** | 结构化代码审查流程 | `code-review-assistant` | ⭐⭐⭐⭐ |
| **7. 部署流程** | 自动化部署和发布 | `deploy-to-prod`、`staging-verify` | ⭐⭐⭐ |
| **8. 团队规范** | 统一团队编码风格 | `team-style-guide`、`commit-message` | ⭐⭐⭐ |
| **9. 调研整理** | 深度调研并生成知识文档 | `research` | ⭐⭐⭐⭐ |

### 8.3 SKILL.md 编写最佳实践

#### 8.3.1 YAML Frontmatter 规范

**必需字段：**

```yaml
---
name: skill-name              # 1-64 字符，小写字母 + 数字 + 连字符
description: 清晰描述技能用途和触发场景  # 1-1024 字符，决定触发准确性
---
```

**可选字段：**

```yaml
---
name: research 文档调研与整理
description: 深度调研任意主题并生成结构化知识文档
aliases: [调研，整理文档，研究]  # 自然语言触发词
triggers: [调研，研究，整理一份，帮我调研]  # 隐式触发词
author: your-name
version: "1.0.0"
license: Apache-2.0
compatibility: Requires git, docker  # 环境要求
metadata:
  team: platform
  category: research
---
```

**description 字段的重要性：**

```
❌ 不好的描述：
description: 处理代码相关任务

✅ 好的描述：
description: 结构化代码审查指南，包含安全检查清单、性能分析、测试覆盖率验证
```

**关键点：** Claude 通过语义理解 description 来判断是否触发 Skill，不是关键词匹配。

#### 8.3.2 内容编写原则

**1. 不要陈述显而易见的事（Don't State the Obvious）**

Claude Code 非常了解代码库，也有丰富的编码知识。

```
❌ 冗余指令：
- "阅读用户指定的文件"
- "理解代码结构"
- "给出你的分析"

✅ 有价值的指令：
- "优先检查是否存在 SQL 注入风险，重点关注字符串拼接的查询语句"
- "使用以下检查清单验证 API 端点的安全性：[具体清单]"
- "输出格式必须遵循团队的安全报告模板（见 references/security-report.md）"
```

**2. 提供可执行的检查清单**

```markdown
## 代码审查清单

### 安全检查
- [ ] 是否存在 SQL 注入风险（重点关注字符串拼接查询）
- [ ] 是否有适当的输入验证
- [ ] 敏感数据是否加密存储

### 性能检查
- [ ] 是否存在 N+1 查询问题
- [ ] 大列表是否有分页处理
- [ ] 是否存在内存泄漏风险
```

**3. 定义清晰的输出格式**

```markdown
## 输出格式规范

审查报告必须包含以下部分：

1. **问题汇总**（按严重程度排序）
2. **具体问题**（文件：行号）
3. **修复建议**（附代码示例）
4. **风险评估**（高/中/低）
```

**4. 提供踩坑清单（Gotchas）**

```markdown
## 常见陷阱

### 陷阱 1：直接字符串拼接 SQL

❌ 错误：
const query = "SELECT * FROM users WHERE id = " + userId;

✅ 正确：
const query = "SELECT * FROM users WHERE id = ?";
db.execute(query, [userId]);

**为什么**：字符串拼接会导致 SQL 注入攻击
```

### 8.4 测试与调试 Skill

#### 8.4.1 验证 Skill 是否生效

**步骤 1：检查可用 Skills**

```bash
# 在 Claude Code 中询问
"What Skills are available?"
```

**步骤 2：使用匹配的触发词测试**

```bash
# 如果 Skill 的 description 包含"调研"，尝试：
"帮我调研一下 React 19 的新特性"
```

**步骤 3：观察 Claude 响应**

如果 Skill 正确触发，Claude 会：
- 引用 Skill 中定义的工作流程
- 使用 Skill 中定义的输出格式
- 访问 Skill 目录下的资源文件

#### 8.4.2 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| **Skill 没有触发** | description 不够清晰 | 重写 description，使其更具体 |
| **Skill 没有加载** | 目录名或 SKILL.md 命名错误 | 确认大小写正确（SKILL.md） |
| **Skill 有错误** | 指令逻辑冲突 | 简化指令，分步骤测试 |
| **多个 Skill 冲突** | 触发条件重叠 | 为每个 Skill 定义更具体的场景 |

#### 8.4.3 迭代开发流程

```
1. 创建最小可用版本（MVP）
   - 只包含核心指令
   - 不包含复杂逻辑

2. 测试触发
   - 使用自然语言触发
   - 使用显式命令触发（/skill-name）

3. 观察行为
   - Claude 是否正确理解意图？
   - 输出是否符合预期格式？

4. 逐步增强
   - 添加更多检查清单
   - 添加资源文件
   - 添加可执行脚本
```

### 8.5 高级用法

#### 8.5.1 与 MCP 配合

**MCP 是"通路"，Skill 是"导航系统"**：

```
MCP 负责：
- 调用外部工具（Google Drive、GitHub、Linear）
- 获取数据

Skill 负责：
- 定义如何使用这些工具
- 提供工作流程和最佳实践
```

**示例：项目冲刺规划 Skill + Linear MCP**

```yaml
name: sprint-planner
description: 使用 Linear MCP 进行项目冲刺规划
---

## 工作流程

1. 通过 Linear MCP 获取当前项目状态
2. 分析团队速度和容量
3. 建议任务优先级
4. 在 Linear 中创建带标签和估算的任务
```

#### 8.5.2 控制 Skill 可见性

**全局 Skill**（所有项目可用）：
```
~/.claude/skills/skill-name/
```

**项目级 Skill**（仅当前项目可用）：
```
.claude/skills/skill-name/
```

### 8.7 Anthropic 内部 9 条经验总结

1. **从"小而专"开始**：避免创建"万能 Skill"，每个 Skill 专注解决 2-3 个具体用例

2. **强调可测试性**：把 Skill 当代码一样管理，编写测试用例

3. **验证类 Skill 价值最高**：即使让一个工程师花一周时间打磨验证 Skill 也是值得的

4. **让 Claude 录制输出**：验证类 Skill 可以让 Claude 录制操作视频，方便审查

5. **强制断言验证**：在流程的每一步强制插入断言，验证状态是否正确

6. **包含踩坑清单**：对于内部库，提供"最容易犯的错误"清单

7. **不要假设唯一性**：Skill 要能和其他 Skill 并存运行

8. **描述决定触发**：description 字段要清晰、具体，有上下文表达力

9. **迭代是关键**：没有完美的第一版，根据实际使用反馈持续改进
