# 文档撰写规范

> 知识库项目 - 内容深度与格式标准

---

## 1. 核心原则

### 1.1 内容深度要求

**每个核心概念必须包含以下 6 个要素：**

| 要素 | 说明 | 示例 |
|------|------|------|
| **概念定义** | 用简洁清晰的语言解释"是什么"和"为什么" | "Fiber 是 React 16+ 的新架构，用于解决大规模组件树的性能问题" |
| **工作原理** | 说明内部机制和运作流程，避免只给代码示例 | 解释 Fiber 如何将渲染任务拆分为可中断的小任务 |
| **源码解析** | 关键特性需附带源码级别的解析或实现思路 | 展示 Fiber 节点的数据结构定义 |
| **代码示例** | 配合文字说明的完整代码示例 | 提供 Fiber 使用的实际代码 |
| **常见误区** | 指出开发者容易误解的地方 | "Fiber 不是虚拟 DOM，而是一种新的架构模式" |
| **最佳实践** | 给出实际开发中的推荐使用方式 | "在性能敏感场景使用 React.memo 配合 Fiber" |

### 1.2 禁止事项

| 禁止项 | 说明 | 反例 |
|--------|------|------|
| ❌ 只给代码不给解释 | 不要只给代码示例而不解释原理 | 只写 `useState(0)` 不解释内部机制 |
| ❌ 只罗列 API 表面用法 | 不要只罗列 API 而不深入底层机制 | 只说"怎么用"不说"为什么" |
| ❌ 遗漏关键概念的来龙去脉 | 必须解释为什么需要这个特性 | 不解释为什么需要 Fiber |
| ❌ 使用模糊词汇 | 不要使用"魔法"等模糊词汇 | "这是 React 的魔法" |

## 2. 文档结构规范

### 2.1 章节编号规则

```markdown
# 一级标题（文档标题）

## 1. 一级章节

### 1.1 二级章节

#### 1.1.1 三级章节（如需要）

## 2. 一级章节
```

**规则：**
- 使用阿拉伯数字编号（1, 2, 3...）
- 子章节使用小数点分隔（1.1, 1.2, 2.1...）
- 章节编号必须连续，无跳跃
- 最多支持三级章节

### 2.2 章节分隔

- 主要章节（## 级别）之间使用 `---` 分隔
- 次要章节（### 级别）之间不需要分隔符

### 2.3 目录结构

- 文档开头可选择性添加目录
- 目录使用 Markdown 链接语法
- 目录与章节必须一一对应

## 3. 代码示例规范

### 3.1 代码块语言标注

````markdown
```javascript
// JavaScript 代码
const [count, setCount] = useState(0);
```

```jsx
// React JSX 代码
function Counter() {
  return <div>{count}</div>;
}
```

```css
/* CSS 代码 */
.container {
  display: flex;
}
```

```bash
# Shell 命令
npm install react
```
````

### 3.2 代码示例必须配有文字说明

**❌ 不好的示例：**

````markdown
```javascript
const [count, setCount] = useState(0);
setCount(count + 1);
```
````

**✅ 好的示例：**

````markdown
**原理**：`useState` 内部维护一个状态链表，每个 Hook 按调用顺序存储在链表中。`setCount` 函数会触发重新渲染。

**代码**：
```javascript
const [count, setCount] = useState(0);
setCount(count + 1);
```

**注意**：`setCount` 是异步的，多次调用会合并更新。如果需要基于之前的状态更新，应使用函数形式：
```javascript
setCount(prevCount => prevCount + 1);
```
````

### 3.3 代码中的注释规范

- 关键逻辑必须添加注释
- 注释使用中文
- 复杂算法需附带思路说明

## 4. 表格规范

### 4.1 表格格式

```markdown
| 表头 1 | 表头 2 | 表头 3 |
|--------|--------|--------|
| 内容 1 | 内容 2 | 内容 3 |
| 内容 4 | 内容 5 | 内容 6 |
```

### 4.2 表格对齐

- 默认左对齐
- 数字列可右对齐（使用 `|---:`）
- 需要居中对齐使用（使用 `|:---:`）

```markdown
| 左对齐 | 右对齐 | 居中对齐 |
|:-------|-------:|:--------:|
| 内容 | 100 | 内容 |
```

## 5. 术语与命名规范

### 5.1 技术名词

- 首次出现时使用 **英文原名（中文翻译）** 格式
- 后续可统一使用中文或英文

**示例：**
- 首次：`useState（React Hook）`
- 后续：`useState` 或 `状态 Hook`

### 5.2 文件路径

- 使用正体代码格式标注
- 相对路径使用 `./` 或 `../` 前缀

**示例：** `.claude/CLAUDE.md`

### 5.3 命令与快捷方式

- Shell 命令使用代码块或行内代码
- Skill 命令使用反引号

**示例：** `/research`、`npm install`

## 6. 引用规范

### 6.1 官方文档引用

- 引用官方文档时必须标注来源
- 重要概念附上官方文档链接

**示例：**

```markdown
> 来源：[React 官方文档 - useState](https://react.dev/reference/react/useState)
```

### 6.2 内部文档引用

- 引用项目内其他文档时使用相对路径

**示例：**

```markdown
详见 [React 核心知识体系.md](../React/React 核心知识体系.md#3-hooks)
```

## 7. 文档检查清单

在文档完成后，必须完成以下检查：

### 7.1 结构检查

- [ ] 章节编号连续且无重复
- [ ] 无重复的章节标题
- [ ] 目录与实际章节对应
- [ ] 章节顺序合理

### 7.2 内容深度检查

- [ ] 每个核心概念包含定义
- [ ] 每个核心概念包含工作原理
- [ ] 关键特性包含源码解析
- [ ] 代码示例配有文字说明
- [ ] 指出常见误区

### 7.3 格式检查

- [ ] Markdown 格式正确
- [ ] 章节分隔符统一
- [ ] 代码块标注语言
- [ ] 表格有表头分隔线

### 7.4 检查命令

```bash
# 检查章节编号
grep "^## " 文档名.md

# 检查重复标题
grep "^## " 文档名.md | sort | uniq -d

# 验证目录链接
# 手动比对目录与实际章节是否一一对应
```

## 8. 文档维护

### 8.1 版本更新

- 文档发生重大变更时更新版本历史
- 版本历史记录在文档末尾

### 8.2 勘误流程

- 发现错误时记录在文档开头或末尾
- 标注勘误日期和内容

### 8.3 评审流程

- 重要文档需经过至少一次评审
- 评审记录保存在文档末尾
