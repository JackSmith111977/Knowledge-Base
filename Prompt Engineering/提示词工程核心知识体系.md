## 1. 概述

### 1.1 什么是提示词工程

**提示词工程**（Prompt Engineering）是针对生成式 AI 模型（如 GPT、Claude、Gemini 等）的输入优化技术，通过设计精准的提示（Prompt）引导模型输出符合预期的结果。

**本质：** 构建模型输入与输出之间的映射规则，解决大模型"理解偏差"问题。

**为什么重要：**
- 即使模型参数规模相同，不同提示方式可能导致输出质量差异超过 **60%**（斯坦福 2023 年研究）
- 在代码生成场景中，有效的提示设计可使模型输出准确率提升 **3 倍**
- 麦肯锡 2024 年报告显示，实施提示工程的企业在客户服务场景中平均减少 **35%** 的人工干预

### 1.2 从 Prompt Engineering 到 Context Engineering

2025 年 AI 应用开发的核心能力已从"如何措辞"转向"提供什么信息"——这正是从 **Prompt Engineering** 到 **Context Engineering** 的本质飞跃。

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|---------------------|
| 关注点 | "怎么写提示词" | "提供什么信息" |
| 范围 | 单次提示优化 | 整个上下文管理系统 |
| 时间维度 | 静态、单次交互 | 动态、多轮对话 |
| 信息源 | 用户输入 | 多源整合（历史、检索、工具） |
| 类比 | 写入 RAM 的指令 | 整个内存管理系统 |

**关键洞察：** Prompt Engineering 是 Context Engineering 的子集。当构建能够自主完成复杂任务的 AI Agent 时，仅优化提示词措辞远远不够——你需要管理对话历史、检索外部知识、调用工具、维护长期记忆。

### 1.3 2025-2026 范式转变

**四大核心转变：**

1. **从一次性查询到迭代对话** — 像 Code Review 一样逐步逼近
2. **从通用策略到模型特定策略** — 不同模型需要不同提示方法
3. **从参数规模到计算时间** — Test-Time Compute 成为新焦点
4. **从经验主义到科学方法** — 系统化测试、可复现性、持续优化

## 2. 核心概念

### 2.1 Tokenization（词元化）

**定义：** LLM 不处理字符，而是处理 Token（词元）。一个 Token 可能是一个完整单词、单词的一部分、标点符号，或几个汉字。

**为什么重要：**
- **成本与限制**：了解不同模型的 Token 限制（Context Window）和计费方式
- **效率**：中文通常比英文更耗费 Token
- **上下文管理**：在 Agent 开发中，如何管理和压缩历史对话，使其不超出 Token 限制是关键工程问题

### 2.2 上下文学习（In-Context Learning）

**定义：** 模型通过提示词中的示例学习任务模式，无需参数更新即可完成新任务。

**三种模式：**

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| Zero-shot | 仅任务描述，无示例 | 简单、常见任务 |
| One-shot | 1 个示例 | 格式要求明确的任务 |
| Few-shot | 2-5 个示例 | 复杂、风格敏感任务 |

### 2.3 模型偏差与幻觉

**模型偏差（Bias）：** LLM 基于训练数据学习，可能存在文化、性别等固有偏差。

**幻觉（Hallucination）：** 模型生成看似合理但虚假的信息。

**应对策略：**
- **去偏提示**：在 Prompt 中明确要求"保持中立、客观，考虑多种观点"
- **幻觉控制**：使用 RAG 架构，或在 Prompt 中强调"仅根据我提供的信息回答"

### 2.4 系统提示词 vs 用户提示词

| 类型 | 类比 | 特点 |
|------|------|------|
| 系统提示词 | 公司规章制度 | 长期有效，定义身份和边界 |
| 用户提示词 | 具体工作任务 | 单次有效，执行具体任务 |

## 3. 快速入门

### 3.1 最小可用提示结构

```
[角色] + [任务] + [约束] + [输出格式]
```

**示例：**

```
你是一位资深 Python 工程师（角色），
需要将以下 JSON 数据转换为 SQL 查询语句（任务），
要求使用 CTE 语法且查询效率最优（约束）。

输入：{"table":"orders", "fields":["id","amount"], "filter":{"status":"completed"}}

输出格式：```sql
-- 完整的 SQL 查询
```
```

### 3.2 立即提升效果的 3 个技巧

**技巧 1：用肯定句描述期望输出**

❌ 差："不要使用省略号"
✅ 好："请使用完整的标点符号，如'......'而非'...'"

**技巧 2：解释约束背后的原因**

❌ 差："永远不要使用专业术语"
✅ 好："请使用简单语言，因为读者是 10 岁的小学生"

**技巧 3：提供示例而非仅描述**

❌ 差："输出要简洁"
✅ 好："输出格式示例：'问题 1: ✅ 正确 | 问题 2: ⚠️ 需改进'"

## 4. 基础方法论

### 4.1 RACE 框架（Anthropic 推荐）

RACE 框架被 OpenAI 官方指南列为首选设计模型：

| 要素 | 说明 | 示例 |
|------|------|------|
| **R**ole（角色） | 定义 AI 的身份和专业领域 | "你是一位资深前端工程师" |
| **A**ction（行动） | 明确要执行的任务 | "请审查以下代码的安全性" |
| **C**ontext（情境） | 提供背景信息和使用场景 | "这是生产环境的支付系统代码" |
| **E**xpectation（期望） | 描述理想的输出格式和质量标准 | "输出包含问题列表、风险等级、修复建议" |

**完整示例：**

```
你是一位资深前端工程师（Role），
请审查以下 React 组件代码的安全性（Action）。
这是生产环境的支付系统代码，需要符合 PCI-DSS 标准（Context）。
输出包含：问题列表、风险等级（高/中/低）、具体修复建议和代码示例（Expectation）。
```

### 4.2 2025-2026 编码专用 7 原则

| 优先级 | 原则（口诀） | 为什么有效 | 典型提升幅度 |
|--------|-------------|-----------|-------------|
| ★★★★★ | 1. 先给完整上下文，再给任务 | AI 没有"读心"能力，缺上下文 ≈ 给缺包的代码让编译 | 50-80% |
| ★★★★★ | 2. 用角色 + 目标 + 约束 + 格式四件套 | 相当于给函数签名 + 注释 + 限制条件 | 基础框架 |
| ★★★★☆ | 3. Chain-of-Thought 必须写"一步一步思考" | 极大降低逻辑错误，尤其算法、复杂业务逻辑 | 逻辑题翻倍 |
| ★★★★☆ | 4. Few-shot 示例 > Zero-shot | 给 1-3 个高质量输入 - 输出对，AI 立马"懂风格" | 风格统一 |
| ★★★★☆ | 5. 明确说不要做什么（负面指令） | 防止 AI 加戏、写冗余代码、引入不想要的依赖 | 减少返工 |
| ★★★☆☆ | 6. 迭代式对话 > 一次性完美 prompt | 先粗后精，像 code review 一样逐步逼近 | 长期最强 |
| ★★★☆☆ | 7. 输出格式用 Markdown + 代码块 + 文件路径 | 方便复制、review、直接贴进 PR | 协作效率 |

### 4.3 结构化设计三原则

**原则 1：核心信息前置**
- 将关键约束放在提示前 20% 位置
- 模型对开头内容的注意力更高

**原则 2：使用分隔符**
- 用 ```、---、### 等分隔不同部分
- 帮助模型清晰识别结构边界

**原则 3：分块加载**
- 超过上下文窗口限制时，分批次输入
- 保持上下文连贯性

## 5. 高级技术

### 5.1 Chain-of-Thought（思维链）

**定义：** 引导模型展示推理过程，而非直接给出答案。

**标准格式：**
```
请一步一步思考：
1. 首先分析问题...
2. 然后...
3. 最后得出结论...
```

**效果：** 逻辑题准确率提升可达 **100%**（Google Research）

### 5.2 Few-Shot Learning（少样本学习）

**设计原则：**
- 示例数量：2-5 个为佳
- 示例质量：覆盖典型场景和边界情况
- 示例格式：输入 - 输出对清晰分隔

**示例：**
```
示例 1:
输入："这个产品很好用"
输出：{"sentiment": "positive", "score": 0.9}

示例 2:
输入："服务态度太差了"
输出：{"sentiment": "negative", "score": 0.85}

示例 3:
输入："快递还可以，就是包装有点破损"
输出：{"sentiment": "neutral", "score": 0.6}

请分析以下文本的情感：
输入："价格有点贵，但质量确实不错"
输出：
```

### 5.3 自我完善回路（Self-Refinement Loop）

**定义：** 引导模型对自身输出进行反思和改进。

**格式：**
```
[任务指令]

完成后，请：
1. 检查你的回答是否满足所有约束条件
2. 识别可能的改进点
3. 输出最终优化版本
```

### 5.4 提示词版本控制与评估

**生产环境最佳实践：**

| 实践 | 说明 |
|------|------|
| **版本控制** | 将关键 Prompt 存储在 Git 中，确保可回溯性 |
| **评估指标** | 定义量化指标（准确率、相关性、格式遵循度） |
| **黄金数据集** | 建立包含输入和期望输出的测试集，自动化测试 |

## 6. Claude 特定最佳实践

### 6.1 Claude 4.x 模型特点

**核心特征：**
- 对明确指令执行力极强
- 更像严谨的程序员，需要清晰的规格说明书
- 模糊请求只会得到平庸的通用回答

**效果对比：**

❌ 差："创建一个分析仪表板"
✅ 好："创建一个分析仪表板。包含尽可能多的相关功能和交互。超越基础功能，创建一个功能完整的实现。"

### 6.2 提供背景与动机

Claude 4.x 具备强大的推理能力，解释"为什么"有助于模型理解深层目标。

❌ 差："永远不要使用省略号"
✅ 好："您的响应将由文本转语音引擎朗读，因此永远不要使用省略号，因为文本转语音引擎不知道如何发音。"

### 6.3 肯定性描述优于否定性禁止

❌ 差："不要使用碎片化的列表"
✅ 好："请使用流畅的散文形式，保持段落连贯性，仅在必要时使用项目符号。"

### 6.4 Claude Code 特定技巧

**适用场景：** 使用 Claude Code 进行软件开发时

| 技巧 | 说明 |
|------|------|
| 文件路径明确 | 始终指定完整文件路径，如 `src/components/Button.tsx` |
| 命令可执行 | 提供完整的 shell 命令，包括必要的参数和标志 |
| 上下文引用 | 引用项目中已有文件，如"参考 `src/utils/api.ts` 的封装模式" |
| 增量修改 | 明确说明是"新增"、"修改"还是"删除" |

## 7. Agent 系统设计

### 7.1 Agent 提示词架构

**核心组件：**

```
System Prompt（系统提示词）
├── 角色定义
├── 能力边界
├── 安全护栏
└── 默认行为规则

User Prompt（用户提示词）
├── 任务描述
├── 上下文信息
├── 约束条件
└── 输出格式
```

### 7.2 工具调用提示设计

**格式：**
```
你有以下工具可用：
- search: 搜索网络信息
- read_file: 读取文件内容
- write_file: 写入文件

当需要调用工具时，请使用以下格式：

```json
{
  "tool": "工具名称",
  "arguments": {
    "参数 1": "值 1",
    "参数 2": "值 2"
  }
}
```

**最佳实践：**
- 明确说明每个工具的用途和限制
- 提供工具调用的成功/失败示例
- 说明多工具调用的顺序依赖

---

### 7.3 多 Agent 协作提示设计

**场景：** 多个专用 Agent 协作完成复杂任务

**架构模式：**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Planner    │ →  │  Executor   │ →  │  Reviewer   │
│  (规划者)    │    │  (执行者)    │    │  (审查者)    │
└─────────────┘    └─────────────┘    └─────────────┘
```

**提示词设计要点：**
- 每个 Agent 有明确的角色边界
- 定义清晰的输入/输出契约
- 包含 Agent 间的信息传递格式

---

### 7.4 上下文管理策略

**核心挑战：** Token 限制下的信息优先级管理

**策略：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **摘要压缩** | 使用 TF-IDF 等算法提取关键特征 | 长对话历史 |
| **核心信息前置** | 将关键约束放在提示前 20% 位置 | 所有场景 |
| **动态加载** | 根据当前任务按需检索相关上下文 | RAG 系统 |
| **记忆分层** | 短期记忆（对话）+ 长期记忆（向量库） | Agent 系统 |

## 8. 常见误区与避坑指南

### 8.1 五大常见陷阱

| 陷阱 | 错误做法 | 正确做法 |
|------|----------|----------|
| **模糊指令** | "帮我优化一下代码" | "请优化以下函数的时间复杂度，目标从 O(n²) 降至 O(n log n)" |
| **缺少上下文** | "这个 bug 怎么修？" | "在 React 18 项目中，useEffect 在组件卸载后仍然触发，代码如下..." |
| **否定式描述** | "不要用复杂的句子" | "请使用简短句，每句不超过 20 个字" |
| **缺少示例** | "输出 JSON 格式" | "输出格式示例：`{"name": "张三", "age": 25}`" |
| **一次性完美主义** | 期望一个 prompt 解决所有问题 | 采用迭代式对话，逐步 refine |

### 8.2 来源单一陷阱

**错误：** 只看官方文档或单一来源

**正确：** 至少 3-5 个来源交叉验证
- 官方文档
- 技术博客
- GitHub Issues
- 社区讨论（Stack Overflow、Reddit）
- 源码分析

### 8.3 忽视模型差异

**错误：** 对所有模型使用相同的提示策略

**正确：** 根据模型特性调整

| 模型类型 | 特点 | 提示策略 |
|----------|------|----------|
| **混合推理架构** (Claude Opus 4.5, Gemini 3 Pro) | 动态调整推理深度 | 明确指定期望深度 |
| **固定推理模型** (O3, DeepSeek-R1) | 内部推理过程固定 | 信任模型，减少干预 |
| **传统快速模型** (Qwen-372B-Instruct) | 快速响应 | 简洁指令，减少 CoT |
