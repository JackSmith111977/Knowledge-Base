## 第 4 章 核心技术栈详解

### 4.1 LLM 基础与 API 调用

**主流大模型对比：**

| 模型 | 提供商 | 特点 | 适用场景 |
|------|--------|------|----------|
| GPT-4/4o | OpenAI | 综合能力最强 | 通用任务 |
| Claude 3.5 | Anthropic | 长上下文、代码能力强 | 文档分析、代码生成 |
| 通义千问 | 阿里 | 中文优化、成本低 | 中文应用 |
| 文心一言 | 百度 | 中文知识丰富 | 中文内容生成 |
| DeepSeek | 深度求索 | 性价比高 | 大规模调用 |

**流式输出封装示例：**
```typescript
async function* streamCompletion(messages: Message[]) {
  const response = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify({ messages, stream: true }),
  });
  
  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    yield decoder.decode(value);
  }
}
```

### 4.2 Prompt 工程进阶

**五要素框架：**
1. **角色定义**：设定 AI 的专业身份
2. **任务说明**：清晰描述要完成的任务
3. **约束条件**：限定输出范围和格式
4. **示例引导**：Few-shot 示例帮助理解
5. **输出格式**：指定 JSON/Markdown 等格式

**思维链（Chain-of-Thought）示例：**
```
请一步步思考，先分析问题，再给出答案。

问题：某公司有 100 名员工，其中 60% 是技术人员...
思考过程：
1. 首先计算技术人员数量：100 × 60% = 60 人
2. ...
答案：...
```

### 4.3 RAG 系统构建

**RAG 完整流程：**
```mermaid
flowchart LR
    A[文档加载] --> B[文本切片]
    B --> C[向量化 Embedding]
    C --> D[存入向量数据库]
    D --> E[用户提问]
    E --> F[问题向量化]
    F --> G[检索相似文档]
    G --> H[检索结果 + 问题送 LLM]
    H --> I[生成答案]
```

**向量数据库选型对比：**

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| Chroma | 轻量级、易上手 | 本地开发、小型项目 |
| Pinecone | 托管服务、性能优 | 生产环境、大规模 |
| Milvus | 开源、功能全 | 企业级部署 |
| Weaviate | 支持混合检索 | 需要关键词 + 向量检索 |

### 4.4 Agent 开发框架

**三大框架对比：**

| 框架 | 语言 | 特点 | 适合场景 |
|------|------|------|----------|
| LangChain | Python/JS | 生态最完善，组件丰富 | 通用 Agent 开发 |
| AutoGen | Python | 多智能体协作 | 复杂任务编排 |
| CrewAI | Python | 角色分工明确 | 团队协作类任务 |
| LangGraph | Python/JS | 图结构工作流 | 状态机类应用 |

### 4.5 Function Calling 与工具调用

**核心原理：**
1. 定义工具 Schema（名称、描述、参数）
2. LLM 判断是否需要调用工具
3. 返回工具调用请求（函数名 + 参数）
4. 执行工具并将结果返回给 LLM
5. LLM 综合工具输出生成最终答案

### 4.6 前端专属 AI 技术

| 技术 | 用途 | 特点 |
|------|------|------|
| TensorFlow.js | 浏览器端 ML 模型 | 无需后端，隐私保护 |
| Transformers.js | 浏览器端 Transformer | 支持 BERT、GPT-2 等 |
| ONNX Runtime Web | 跨平台模型推理 | 性能优化、多格式支持 |
| WebLLM | 浏览器端 LLM | 本地运行 7B 模型 |

## 第 8 章 常见误区与避坑指南

### 8.1 三大致命误区

#### 误区 1：重学 Python 放弃 JS 生态

❌ **错误做法：**
- 认为做大模型必须精通 Python
- 放弃已有的 JavaScript/TypeScript 技术栈
- 从零开始学 Python 生态

✅ **正确做法：**
- 利用 LangChain.js（官方维护的 JavaScript 版本）
- 使用 Transformers.js 在浏览器运行 BERT 等模型
- Python 仅学习 API 调用基础，不追求精通

**对比表：**

| 方案 | 学习成本 | 产出时间 | 推荐度 |
|------|----------|----------|--------|
| 放弃 JS 重学 Python | 6-12 个月 | 1 年+ | ❌ 不推荐 |
| JS 为主 + Python 辅助 | 2-3 个月 | 3-6 个月 | ✅ 推荐 |

#### 误区 2：只做调用层不碰核心逻辑

❌ **错误做法：**
- 满足于调 API 生成 Demo
- 不理解 RAG、Agent 的内部原理
- 无法解决生产环境的复杂问题

✅ **正确做法：**
- 深入理解 RAG 检索优化（混合检索、重排序）
- 掌握 Agent 任务编排（ReAct、ToT 框架）
- 学习性能与成本控制（Token 消耗监控、缓存策略）

#### 误区 3：忽视工程化能力

❌ **错误做法：**
- 只关注 AI 模型能力
- 忽视错误处理、日志监控、A/B 测试
- 无法将 Demo 转化为生产级应用

✅ **正确做法：**
- 发挥前端工程化优势（CI/CD、容器化、监控）
- 建立 AI 应用的测试体系（Prompt 测试、端到端验证）
- 设计优雅降级和容错机制

### 8.3 转型风险控制与备选方案

**转型失败的风险因素：**

2. 缺乏项目实战（至少完成 2-3 个完整项目）
