## 7.1 案例一：电商后台系统 AI 化改造

### 7.1.1 项目背景

**基本信息：**
- **团队规模：** 6 人开发团队
- **技术栈：** Java + Spring Boot 后端，Vue 3 前端
- **代码规模：** 约 50 万行
- **项目历史：** 3 年维护历史

**痛点：**
1. 新人上手慢（平均 4 周才能独立开发）
2. 需求变更频繁，返工率高（35%）
3. 文档缺失率 41%，代码与文档脱节
4. 代码审查耗时，PR 平均等待 9.6 天

### 7.1.2 迁移过程

**第 1-2 周：基础建设**
- 建立 `.claude/` 目录，编写 `CLAUDE.md`
- 配置 Claude Code 和 MCP 服务
- 创建 `docs/` 目录结构
- 选择用户模块作为试点

**第 3-4 周：小范围试点**
- 执行完整的 SDD 工作流
- 为用户管理功能创建 Spec 文档
- AI 生成代码和测试
- 收集反馈并调整流程

**第 5-8 周：全面推广**
- 将试点经验推广至全项目
- 为每个功能建立 Spec 文档
- 建立 Spec→代码→测试的闭环
- 组织团队成员培训

**第 9-12 周：持续优化**
- 根据反馈优化 Spec 模板
- 建立度量指标
- 持续改进流程

### 7.1.3 改造效果

| 指标 | 改造前 | 改造后 | 提升 |
|------|--------|--------|------|
| **需求交付周期** | 5.2 天 | 1.8 天 | -65% |
| **返工率** | 35% | 8% | -77% |
| **新人上手时间** | 4 周 | 2 周 | -50% |
| **文档缺失率** | 41% | 0% | -100% |
| **PR 等待时间** | 9.6 天 | 2.4 天 | -75% |
| **测试覆盖率** | 32% | 87% | +172% |

### 7.1.4 关键成功因素

1. **管理层支持** — CTO 亲自推动，给予充分资源
2. **试点先行** — 选择独立模块试点，降低风险
3. **培训到位** — 组织 3 场培训，确保团队掌握
4. **度量驱动** — 建立明确的度量指标，持续跟踪
5. **文化变革** — 从"代码为王"转向"文档优先"

### 7.1.5 踩坑与解决方案

| 坑 | 问题描述 | 解决方案 |
|----|----------|----------|
| **Spec 写得太细** | AI 失去发挥空间，代码僵化 | 只写"做什么"，不写"怎么做" |
| **文档更新滞后** | 代码改了文档没改 | 将文档更新纳入 DoD |
| **团队抵触** | 老员工习惯手写代码 | 设立 AI Champion，带动氛围 |
| **AI 生成代码质量不稳定** | 有时生成低质量代码 | 加强 Spec 审查，增加测试覆盖 |

## 7.2 案例二：个人开发者从 0 开发太阳系可视化

### 7.2.1 项目背景

**基本信息：**
- **开发者：** 个人开发者，天体物理专业背景
- **目标：** 3D 太阳系可视化互动页面
- **技术栈：** React + Three.js

### 7.2.2 关键教训

**踩坑：**
- 一开始选择 React Native 跨平台方案
- 工程配置复杂，AI 陷入编译问题循环
- 3 周时间过去，核心功能还未开始

**调整：**
- 改为纯 Web 方案（React + Three.js）
- 使用 Vite 快速搭建项目
- 2 周内完成核心功能上线

### 7.2.3 文档优先实践

**Spec 文档结构：**

```markdown
# 太阳系可视化 Spec

## 1. 功能目标
- 展示太阳系八大行星的 3D 模型
- 支持用户交互（缩放、旋转）
- 显示行星基本信息

## 2. In Scope
- 3D 行星模型渲染
- 轨道动画
- 信息面板

## 3. Out Scope
- 多人协作
- 用户账号系统
- 数据存储
```

**效果：**
- Spec 确认后，AI 按步骤执行
- 2 周完成核心功能
- 代码质量高，测试覆盖完整

### 7.2.4 最佳实践

1. **选择 AI 熟悉的技术栈** — React + Three.js 训练数据丰富
2. **避免过度设计** — 先跑通核心功能，再考虑扩展
3. **用文档记录关键决策** — 方便后续调整

## 7.3 案例三：OpenAI 内部 Harness 实践

### 7.3.1 项目背景

**基本信息：**
- **团队规模：** 3 人（后扩展到 7 人）
- **时间：** 5 个月
- **产出：** 100 万行代码，1500 个 PR
- **规则：** 人类不手写一行代码，全部由 Codex Agent 生成

### 7.3.2 关键实践

**1. AGENTS.md 文档**
- 100 行的索引文档
- 包含架构图、设计规范和执行计划入口
- 服务对象是 Agent 而不是人类

**2. 关键知识只能活在代码库里**
- 禁止散落在 Slack 消息或口口相传
- 确保 AI 可以访问所有必要信息

**3. 给 Agent 装上眼睛**
- 集成 Chrome DevTools
- AI 自己截图验证 UI 渲染结果
- 形成自我纠错闭环

**4. 选择"无聊"的技术栈**
- 训练数据中出现越多的库，AI 理解越准确
- 避免使用冷门的新库

**5. "垃圾回收"机制**
- 后台运行周期性 Agent
- 定期扫描代码库里的技术债
- 自动生成清理建议

### 7.3.3 效果数据

| 指标 | 数据 |
|------|------|
| **代码量** | 100 万行 |
| **PR 数量** | 1500 个 |
| **人均日 PR** | 3.5 个 |
| **效率提升** | 约 10 倍 |
| **测试覆盖率** | 91.4% |

## 7.4 最佳实践清单

### 7.4.1 文档编写

- [ ] Spec 文档必须包含 In/Out Scope
- [ ] 验收标准要可测量、可测试
- [ ] 变更记录要详细，包含"为什么变更"
- [ ] 文档要有人类 Sign-off 确认

### 7.4.2 AI 协作

- [ ] 给 AI 足够的上下文（文档 + 代码）
- [ ] 让 AI 主动提问澄清模糊点
- [ ] 分步执行，每步确认后再继续
- [ ] 重要决策要 Sign-off，不口头确认

### 7.4.3 代码审查

- [ ] AI 先自查，生成变更摘要
- [ ] 人类审查 Diff，关注核心逻辑
- [ ] 测试用例要覆盖边界情况
- [ ] 审查通过后再合并

### 7.4.4 知识管理

- [ ] 建立项目知识库
- [ ] 记录踩坑和解决方案
- [ ] 定期回顾，更新最佳实践
- [ ] 新人培训材料要完善
