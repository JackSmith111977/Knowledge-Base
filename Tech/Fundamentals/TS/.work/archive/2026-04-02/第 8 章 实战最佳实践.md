## 8.1 避免 any 的最佳实践（unknown 替代方案）

### 8.1.1 概念定义

**any 类型** 是 TypeScript 中的"顶级类型"（Top Type），可以表示任何值。使用 any 会完全关闭类型检查，使 TypeScript 退化为 JavaScript。

**unknown 类型** 是 TypeScript 3.0 引入的类型安全版本的 any，同样可以表示任何值，但强制要求在使用前进行类型检查或断言。

**核心区别：**

| 特性 | any | unknown |
|------|-----|---------|
| 类型检查 | ❌ 完全禁用 | ✅ 强制检查 |
| 可直接访问属性 | ✅ 是 | ❌ 否 |
| 可赋值给任意类型 | ✅ 是 | ❌ 需断言 |
| 类型安全性 | 低 | 高 |
| 使用场景 | 遗留代码迁移 | 处理未知数据 |

**为什么避免 any：**
1. **丧失智能提示**：IDE 无法提供自动补全
2. **静默失败**：类型错误推迟到运行时
3. **污染传播**：any 会传染给其他类型
4. **违背初衷**：失去 TypeScript 的核心价值

### 8.1.2 工作原理

**any 类型的行为：**

```typescript
let value: any = 42;

// ✅ 所有操作都允许，编译器不检查
value = "hello";
value = { key: "value" };
value.nonExistentMethod();  // 编译通过，运行时可能报错
value.toFixed(2);           // 编译通过，运行时可能报错
```

**unknown 类型的行为：**

```typescript
let value: unknown = 42;

// ❌ 直接操作会报错
// value.toFixed(2);  // TS2531: Object is of type 'unknown'

// ✅ 方式 1：类型收窄（Narrowing）
if (typeof value === "number") {
  value.toFixed(2);  // ✅ 安全
}

// ✅ 方式 2：类型断言
(value as number).toFixed(2);

// ✅ 方式 3：类型守卫函数
function isNumber(v: unknown): v is number {
  return typeof v === "number";
}
if (isNumber(value)) {
  value.toFixed(2);  // ✅ 安全
}
```

**类型收窄机制：**

TypeScript 使用控制流分析（Control Flow Analysis）来跟踪类型变化：

```typescript
function processInput(input: unknown) {
  // input 类型：unknown
  
  if (typeof input === "string") {
    // 在此分支内，input 类型被收窄为 string
    console.log(input.toUpperCase());  // ✅
  }
  
  // 回到这里，input 仍然是 unknown
  // input.toUpperCase();  // ❌
}
```

### 8.1.3 源码/底层解析

**TypeScript 类型检查源码位置：**

TypeScript 的类型检查逻辑位于 `src/compiler/checker.ts` 文件中。

**unknown 类型检查逻辑（简化版）：**

```typescript
// TypeScript 源码逻辑（简化）
function checkPropertyAccessExpression(
  node: PropertyAccessExpression,
  type: Type
) {
  // 检查是否为 unknown 类型
  if (type.flags & TypeFlags.Unknown) {
    // unknown 类型不允许直接访问属性
    error(
      node,
      Diagnostic.Object_is_of_type_unknown
    );
    return;
  }
  
  // any 类型允许访问任何属性
  if (type.flags & TypeFlags.Any) {
    return getAnyType();
  }
  
  // 正常类型检查流程
  return checkPropertyAccess(type, node.name);
}
```

**类型收窄的底层实现：**

```typescript
// 控制流分析核心逻辑
function narrowTypeByCondition(
  type: Type,
  condition: string
): Type {
  // unknown 经过 typeof 检查后被收窄
  if (type.flags & TypeFlags.Unknown) {
    if (condition === "string") {
      return getStringType();
    }
    if (condition === "number") {
      return getNumberType();
    }
    // ...
  }
  
  // any 类型不会被收窄
  if (type.flags & TypeFlags.Any) {
    return type;  // 保持 any
  }
  
  // 联合类型的收窄
  if (type.flags & TypeFlags.Union) {
    return narrowUnionType(type, condition);
  }
  
  return type;
}
```

### 8.1.4 代码示例

**场景 1：处理外部 API 数据**

```typescript
// ==================== 错误示例：使用 any ====================
async function fetchUserData(userId: number): Promise<any> {
  const response = await fetch(`/api/users/${userId}`);
  return response.json();  // 返回 any，调用方无法获得类型安全
}

// 调用方
const user = await fetchUserData(1);
console.log(user.nmae);  // ❌ 拼写错误，但编译器不报错
console.log(user.address.city);  // ❌ 属性不存在，编译器不报错
```

```typescript
// ==================== 正确示例：使用 unknown ====================
interface User {
  id: number;
  name: string;
  email?: string;
}

async function fetchUserData(userId: number): Promise<unknown> {
  const response = await fetch(`/api/users/${userId}`);
  return response.json();  // 返回 unknown
}

// 调用方需要进行类型检查
const result = await fetchUserData(1);

// 方式 1：类型守卫
function isUser(data: unknown): data is User {
  return (
    typeof data === "object" &&
    data !== null &&
    "id" in data &&
    "name" in data &&
    typeof (data as User).id === "number" &&
    typeof (data as User).name === "string"
  );
}

if (isUser(result)) {
  console.log(result.name);  // ✅ 类型安全
}

// 方式 2：try-catch + 断言
try {
  const user = result as User;
  console.log(user.name);
} catch (error) {
  console.error("无效的用户数据");
}
```

**场景 2：JSON 解析函数**

```typescript
// ==================== 安全解析器 ====================
class SafeJSONParser {
  /**
   * 安全解析 JSON 字符串
   * @param json JSON 字符串
   * @returns 解析结果（unknown 类型）
   */
  static parse(json: string): unknown {
    try {
      return JSON.parse(json);
    } catch (error) {
      throw new Error(`JSON 解析失败：${error instanceof Error ? error.message : "未知错误"}`);
    }
  }

  /**
   * 解析并验证类型
   * @param json JSON 字符串
   * @param validator 验证函数
   * @returns 验证后的类型
   */
  static parseWithValidation<T>(
    json: string,
    validator: (data: unknown) => data is T
  ): T {
    const data = this.parse(json);
    if (!validator(data)) {
      throw new Error("数据验证失败");
    }
    return data;
  }
}

// 使用示例
interface Config {
  apiUrl: string;
  timeout: number;
  debug?: boolean;
}

function isConfig(data: unknown): data is Config {
  return (
    typeof data === "object" &&
    data !== null &&
    "apiUrl" in data &&
    "timeout" in data &&
    typeof (data as Config).apiUrl === "string" &&
    typeof (data as Config).timeout === "number"
  );
}

// 解析并验证
const configJson = '{"apiUrl": "https://api.example.com", "timeout": 5000}';
const config = SafeJSONParser.parseWithValidation(configJson, isConfig);
console.log(config.apiUrl);  // ✅ 类型安全
```

**场景 3：通用事件处理器**

```typescript
// ==================== 类型安全的事件系统 ====================
type EventHandler<T = unknown> = (data: T) => void;

interface EventMap {
  userLogin: { userId: number; timestamp: number };
  userLogout: { userId: number };
  error: { message: string; code: number };
}

class TypedEventEmitter {
  private handlers: Map<string, Set<EventHandler>> = new Map();

  on<K extends keyof EventMap>(
    event: K,
    handler: EventHandler<EventMap[K]>
  ): void {
    const handlers = this.handlers.get(event) || new Set();
    handlers.add(handler as EventHandler);
    this.handlers.set(event, handlers);
  }

  emit<K extends keyof EventMap>(
    event: K,
    data: EventMap[K]
  ): void {
    const handlers = this.handlers.get(event);
    handlers?.forEach(handler => handler(data));
  }
}

// 使用示例
const emitter = new TypedEventEmitter();

emitter.on("userLogin", (data) => {
  console.log(`用户 ${data.userId} 登录`);  // ✅ 类型安全
});

emitter.emit("userLogin", { userId: 1, timestamp: Date.now() });
```

### 8.1.5 常见误区

**误区 1：认为 unknown 和 any 可以互换**

```typescript
// ❌ 错误：unknown 不能直接赋值给其他类型
let u: unknown = "hello";
let s: string = u;  // TS2322 错误

// ✅ 正确：需要类型断言或收窄
let s: string = u as string;
if (typeof u === "string") {
  let s: string = u;  // ✅ 收窄后可以
}
```

**误区 2：滥用类型断言绕过检查**

```typescript
// ❌ 错误：使用 as any 绕过类型检查
function process(data: unknown) {
  (data as any).someMethod();  // 失去了类型安全
}

// ✅ 正确：使用类型守卫
function process(data: unknown) {
  if (hasSomeMethod(data)) {
    data.someMethod();  // ✅ 类型安全
  }
}
```

**误区 3：在函数返回类型中使用 any**

```typescript
// ❌ 错误：返回 any 污染调用方
function getData(): any {
  return { id: 1, name: "test" };
}
const data = getData();
data.nonExistent();  // ❌ 编译器不报错

// ✅ 正确：返回具体类型或 unknown
function getData(): { id: number; name: string } {
  return { id: 1, name: "test" };
}
```

**误区 4：在泛型中默认使用 any**

```typescript
// ❌ 错误：泛型默认 any 失去类型约束
function identity<T = any>(arg: T): T {
  return arg;
}
const result = identity("hello");  // T 被推断为 any

// ✅ 正确：泛型默认 unknown
function identity<T = unknown>(arg: T): T {
  return arg;
}
```

### 8.1.6 最佳实践

**1. 严格模式配置**

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,           // 启用所有严格检查
    "noImplicitAny": true,    // 禁止隐式 any
    "useUnknownInCatchVariables": true,  // catch 变量为 unknown
    "noUncheckedIndexedAccess": true     // 索引访问返回 undefined
  }
}
```

**2. unknown 优先原则**

```typescript
// ✅ 推荐：使用 unknown 处理不确定类型
function safeParse(json: string): unknown {
  return JSON.parse(json);
}

// ⚠️ 仅在必要时使用 any
// 场景：迁移遗留代码、编写底层库
function legacyFunction(arg: any): any {
  // ...
}
```

**3. 类型守卫函数封装**

```typescript
// ✅ 创建可复用的类型守卫
function isObject(data: unknown): data is Record<string, unknown> {
  return typeof data === "object" && data !== null;
}

function isString(data: unknown): data is string {
  return typeof data === "string";
}

function isNumber(data: unknown): data is number {
  return typeof data === "number" && !isNaN(data);
}

// 组合使用
function process(data: unknown) {
  if (isObject(data) && isString(data.name)) {
    console.log(data.name);  // ✅ 类型安全
  }
}
```

**4. 使用工具类型处理 unknown**

```typescript
// 定义类型验证器工具
type Validator<T> = (data: unknown) => data is T;

// 组合验证器
function combineValidators<T>(
  ...validators: Validator<T>[]
): Validator<T> {
  return (data): data is T => {
    return validators.every(v => v(data));
  };
}

// 使用
const isValidUser = combineValidators(isObject, hasName, hasEmail);
```

**5. 第三方库类型处理**

```typescript
// 为无类型的库创建声明文件
// src/types/legacy-lib.d.ts
declare module "legacy-lib" {
  export function helper(arg: string): number;
  // 而不是导出 any
}

// 或使用类型断言限制范围
import legacy from "legacy-lib";
const result = (legacy.helper as (arg: string) => number)("test");
```

## 8.2 类型守卫模式（typeof、instanceof、自定义守卫）

### 8.2.1 概念定义

**类型守卫（Type Guard）** 是一种在运行时检查变量类型的表达式或函数，TypeScript 根据检查结果在编译时收窄变量类型。

**三种主要类型守卫：**

1. **typeof 守卫**：检查基本类型（string、number、boolean、symbol、bigint、function、object）
2. **instanceof 守卫**：检查对象是否为某个类的实例
3. **自定义类型守卫**：使用类型谓词（type predicate）定义复杂的类型检查逻辑

**类型谓词（Type Predicate）**：一种特殊的返回类型，语法为 `parameterName is Type`，用于自定义类型守卫函数。

**为什么需要类型守卫：**
- 处理联合类型时安全地访问特定类型的属性
- 在运行时验证外部数据
- 替代不安全的类型断言（as）

### 8.2.2 工作原理

**typeof 守卫机制：**

```typescript
function formatInput(input: string | number): string {
  if (typeof input === "string") {
    // 在此分支内，input 类型被收窄为 string
    return input.trim().toUpperCase();
  }
  // 在此分支内，input 类型被收窄为 number
  return input.toFixed(2);
}
```

**instanceof 守卫机制：**

```typescript
class ApiError extends Error {
  constructor(
    message: string,
    public statusCode: number
  ) {
    super(message);
  }
}

class NetworkError extends Error {
  constructor(
    message: string,
    public retryable: boolean
  ) {
    super(message);
  }
}

function handleError(error: Error): void {
  if (error instanceof ApiError) {
    // error 类型被收窄为 ApiError
    console.log(`API 错误：${error.statusCode}`);
  } else if (error instanceof NetworkError) {
    // error 类型被收窄为 NetworkError
    if (error.retryable) {
      console.log("可重试的网络错误");
    }
  }
}
```

**自定义类型守卫机制：**

```typescript
interface Cat {
  meow(): void;
  climbTrees: boolean;
}

interface Dog {
  bark(): void;
  fetchItems: number;
}

// 自定义类型守卫函数
function isCat(animal: Cat | Dog): animal is Cat {
  return (animal as Cat).climbTrees !== undefined;
}

function trainAnimal(animal: Cat | Dog) {
  if (isCat(animal)) {
    // animal 类型被收窄为 Cat
    animal.meow();
    animal.climbTrees = true;
  } else {
    // animal 类型被收窄为 Dog
    animal.bark();
    console.log(`拾取 ${animal.fetchItems} 个物品`);
  }
}
```

### 8.2.3 源码/底层解析

**TypeScript 类型收窄源码位置：**

TypeScript 的类型收窄逻辑位于 `src/compiler/checker.ts` 和 `src/compiler/narrowing.ts` 文件中。

**typeof 守卫的源码逻辑（简化版）：**

```typescript
// TypeScript 源码（简化）
function narrowTypeByTypeof(
  type: Type,
  expression: Expression,
  literalType: Type
): Type {
  // 处理联合类型
  if (type.flags & TypeFlags.Union) {
    const unionType = type as UnionType;
    const narrowedTypes: Type[] = [];

    for (const constituentType of unionType.types) {
      // 检查每个成员类型是否匹配 typeof 结果
      if (isTypeCompatibleWithLiteralType(constituentType, literalType)) {
        narrowedTypes.push(constituentType);
      }
    }

    return createUnionType(narrowedTypes);
  }

  return type;
}

// typeof 结果映射
function getLiteralTypeForTypeofResult(
  result: string
): Type {
  switch (result) {
    case "string":
      return getStringType();
    case "number":
      return getNumberType();
    case "boolean":
      return getBooleanType();
    case "object":
      // 注意：typeof null === "object"，需要特殊处理
      return getObjectType();
    case "function":
      return getFunctionType();
    case "undefined":
      return getUndefinedType();
    case "bigint":
      return getBigIntType();
    case "symbol":
      return getSymbolType();
  }
}
```

**类型谓词的源码处理：**

```typescript
// 解析类型谓词
function parseTypePredicate(
  typeNode: TypeNode
): TypePredicate | undefined {
  // 识别 "param is Type" 语法
  if (isTypePredicateNode(typeNode)) {
    return {
      parameterName: typeNode.parameterName,
      type: getTypeFromTypeNode(typeNode.type),
      asserts: false
    };
  }

  // 识别 "asserts param is Type" 语法（TypeScript 3.7+）
  if (isAssertsKeyword(typeNode)) {
    return {
      parameterName: typeNode.parameterName,
      type: getTypeFromTypeNode(typeNode.type),
      asserts: true
    };
  }

  return undefined;
}

// 应用类型谓词到控制流
function applyTypePredicateToFlow(
  predicate: TypePredicate,
  flowNode: FlowNode
): void {
  if (predicate.asserts) {
    // asserts 模式：如果谓词为 false，抛出异常
    // 类型在后续代码中被收窄
    flowNode.type = FlowFlags.Throws;
  } else {
    // is 模式：在 true 分支收窄类型
    flowNode.type = FlowBranch;
  }
}
```

**控制流图（CFG）与类型收窄：**

TypeScript 使用控制流分析来跟踪类型变化：

```typescript
function example(value: string | number | null) {
  // 控制流分析：
  // 入口：value 类型为 string | number | null
  
  if (value === null) {

  }
  
  // 分支 1 之后：value 类型为 string | number

  // 分支 2 之后：value 类型为 number

}
```

### 8.2.4 代码示例

**场景 1：可辨识联合（Discriminated Unions）**

### 8.2.5 常见误区

**误区 1：typeof null 返回 "object"**

```typescript
// ❌ 错误：忽略 typeof null 的边界情况
function checkValue(value: unknown) {
  if (typeof value === "object") {
    // value 可能是 null！
    console.log(value.toString());  // 运行时错误：Cannot read property of null
  }
}

// ✅ 正确：先检查 null
function checkValue(value: unknown) {
  if (typeof value === "object" && value !== null) {
    console.log(value.toString());  // ✅ 安全
  }
}
```

**误区 2：instanceof 不能用于接口**

```typescript
interface Point {
  x: number;
  y: number;
}

// ❌ 错误：接口在运行时不存在，不能使用 instanceof
function isPoint(value: unknown): value is Point {
  return value instanceof Point;  // 编译错误！
}

// ✅ 正确：使用 in 操作符检查属性
function isPoint(value: unknown): value is Point {
  return (
    typeof value === "object" &&
    value !== null &&
    "x" in value &&
    "y" in value
  );
}
```

**误区 3：类型守卫不覆盖所有情况**

```typescript
type Result = string | number | null;

// ❌ 不完整：遗漏 null 情况
function process(result: Result) {
  if (typeof result === "string") {
    return result;
  }
  if (typeof result === "number") {
    return result;
  }
  // 遗漏 null 处理
  return result.toString();  // ❌ 可能为 null
}

// ✅ 完整：处理所有情况
function process(result: Result) {
  if (result === null) {
    return "默认值";
  }
  if (typeof result === "string") {
    return result;
  }
  return result.toString();
}
```

**误区 4：在守卫中使用副作用操作**

```typescript
// ❌ 不推荐：守卫函数包含副作用
let callCount = 0;
function isString(value: unknown): value is string {
  callCount++;  // 副作用
  return typeof value === "string";
}

// ✅ 推荐：守卫函数应该是纯函数
function isString(value: unknown): value is string {
  return typeof value === "string";
}
```

## 8.3 第三方库类型扩展（DefinitelyTyped、@types）

### 8.3.1 概念定义

**DefinitelyTyped** 是 TypeScript 生态系统中最重要的基础设施之一，它是一个由社区维护的开源仓库，包含数千个 JavaScript 库的 TypeScript 类型定义文件（.d.ts）。

**@types 包** 是 DefinitelyTyped 项目发布到 npm 的类型定义包，命名格式为 `@types/package-name`。

**核心概念：**
- **类型定义文件（.d.ts）**：描述 JavaScript 代码结构的特殊文件，不生成运行时代码
- **模块扩展（Module Augmentation）**：在已有模块类型定义基础上添加新类型
- **声明合并（Declaration Merging）**：TypeScript 将同名声明合并为一个
- **全局类型扩展**：扩展全局作用域中的类型

**为什么需要类型扩展：**
- 为无类型定义的第三方库添加类型支持
- 修正官方类型定义中的错误
- 扩展库的功能并添加对应类型
- 为项目添加全局类型工具

**声明合并的底层实现：**

```typescript
// TypeScript 源码（简化）
function mergeDeclarations(
  existing: Declaration,
  incoming: Declaration
): Declaration {
  // 接口合并
  if (existing.kind === SyntaxKind.InterfaceDeclaration) {
    return mergeInterfaces(existing, incoming);
  }
  
  // 命名空间合并
  if (existing.kind === SyntaxKind.NamespaceDeclaration) {
    return mergeNamespaces(existing, incoming);
  }
  
  // 函数重载合并
  if (existing.kind === SyntaxKind.FunctionDeclaration) {
    return mergeFunctionOverloads(existing, incoming);
  }
  
  return incoming;
}

// 接口合并逻辑
function mergeInterfaces(
  target: InterfaceDeclaration,
  source: InterfaceDeclaration
): InterfaceDeclaration {
  // 合并成员（不重复添加）
  for (const member of source.members) {
    if (!target.members.some(m => m.name === member.name)) {
      target.members.push(member);
    }
  }
  return target;
}
```

## 8.4 大型项目类型架构设计（monorepo 类型共享）

### 8.4.1 概念定义

**Monorepo（单体仓库）** 是一种将多个项目/包存储在单一仓库中的代码组织方式，TypeScript 类型架构设计在 monorepo 中尤为重要。

**核心概念：**
- **项目引用（Project References）**：TypeScript 3.0+ 支持的多项目编译机制
- **复合项目（Composite Projects）**：使用 `"composite": true` 配置的项目
- **类型共享包**：集中管理跨项目共享的类型定义
- **增量编译**：利用 `.tsbuildinfo` 缓存加速构建

**为什么需要类型架构设计：**
- 避免类型定义重复
- 确保跨项目类型一致性
- 优化编译性能
- 管理复杂的依赖关系

### 8.4.2 工作原理

**TypeScript 项目引用机制：**

```json
// 根 tsconfig.json
{
  "files": [],
  "references": [
```

### 8.4.3 源码/底层解析

**TypeScript 项目引用源码位置：**

TypeScript 的项目引用逻辑位于 `src/compiler/builder.ts` 和 `src/compiler/watch.ts` 文件中。

**项目引用解析流程：**

```typescript
// TypeScript 源码（简化）
function resolveProjectReferences(
  options: CompilerOptions,
  host: CompilerHost
): ResolvedProjectReference[] {
  const references: ResolvedProjectReference[] = [];

  for (const ref of options.references || []) {
    // 1. 解析项目路径
    const configPath = resolveConfigPath(ref.path);

    // 2. 读取 tsconfig.json
    const config = readConfigFile(configPath, host.readFile);

    // 3. 验证是否为复合项目
    if (!config.options.composite) {
      error("引用的项目必须启用 composite");
      continue;
    }

    // 4. 递归解析嵌套引用
    const nestedRefs = resolveProjectReferences(
      config.options,
      host
    );

    references.push({
      config,
      path: configPath,
      references: nestedRefs
    });
  }

  return references;
}
```

**增量编译机制：**

```typescript
// 增量编译核心逻辑
function buildIncremental(
  program: Program,
  oldProgram: Program | undefined,
  infoFile: TsBuildInfo
): Program {
  // 1. 读取缓存
  const cached = readBuildInfo(infoFile);

  // 2. 比较文件变更
  const changedFiles = compareFiles(
    program.getSourceFiles(),
    cached.sourceFiles
  );

  // 3. 重新编译变更的文件
  for (const file of changedFiles) {
    recompile(file);
  }

  // 4. 更新缓存
  writeBuildInfo(infoFile, program);

  return program;
}
```

**声明文件生成逻辑：**

```typescript
// 生成 .d.ts 文件
function emitDeclarationFiles(
  program: Program,
  host: EmitHost
): void {
  for (const sourceFile of program.getSourceFiles()) {
    // 跳过非 TypeScript 文件
    if (!isTypescriptFile(sourceFile)) {
      continue;
    }

    // 生成类型定义
    const declarations = generateDeclarations(sourceFile);

    // 写入文件
    writeFile(
      getDeclarationPath(sourceFile),
      declarations
    );
  }
}
```

### 8.4.5 常见误区

**误区 1：忘记启用 composite 配置**

```json
// ❌ 错误：项目引用要求 composite: true
{
  "compilerOptions": {
    "declaration": true
    // 缺少 composite: true
  }
}

// ✅ 正确
{
  "compilerOptions": {
    "composite": true,
    "declaration": true,
    "declarationMap": true
  }
}
```

**误区 2：循环依赖问题**

```
// ❌ 错误：循环依赖
packages/core 依赖于 packages/app
packages/app 依赖于 packages/core

// ✅ 正确：单向依赖
packages/types (无依赖)
       ↓
packages/core (依赖 types)
       ↓
packages/app (依赖 types, core)
```

**误区 3：忽略 tsBuildInfoFile 配置**

```json
// ❌ 不推荐：使用默认缓存路径
{
  "compilerOptions": {
    "incremental": true
    // 缓存文件散落在各处
  }
}

// ✅ 推荐：统一缓存路径
{
  "compilerOptions": {
    "incremental": true,
    "tsBuildInfoFile": "./dist/.tsbuildinfo"
  }
}
```

**误区 4：在共享类型中包含实现代码**

```typescript
// ❌ 错误：类型包包含实现

// ✅ 正确：仅导出类型
// packages/types/src/common.ts

  debug: boolean;
```

## 本章小结

本章深入讲解了 TypeScript 实战中的核心最佳实践：

1. **避免 any**：使用 unknown 替代，配合类型守卫确保类型安全
2. **类型守卫模式**：掌握 typeof、instanceof、自定义守卫和可辨识联合
3. **第三方库类型扩展**：使用 DefinitelyTyped、@types 包和模块扩展
4. **大型项目类型架构**：monorepo 类型共享、项目引用、增量编译

**核心要点：**
- 永远优先使用 unknown 而非 any
- 使用类型守卫替代类型断言
- 为第三方库创建类型扩展而非修改源码
- monorepo 项目使用复合项目和项目引用管理类型共享
