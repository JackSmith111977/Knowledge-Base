# 第 6 章：工程化实践

> TypeScript 的工程化实践涵盖了从项目配置、类型声明到代码规范的完整开发流程。本章将深入探讨 tsconfig.json 核心配置、.d.ts 声明文件编写、模块声明与增强以及 ESLint 集成等关键主题，帮助开发者构建专业级的 TypeScript 项目。

## 6.1 tsconfig.json 核心配置详解

### 6.1.1 什么是 tsconfig.json？

**概念定义**：`tsconfig.json` 是 TypeScript 项目的配置文件，它定义了编译选项、包含/排除的文件、项目引用等关键信息。

**为什么需要 tsconfig.json**：
1. **项目标识**：告诉 TypeScript 编译器当前目录是一个 TS 项目
2. **编译控制**：精确控制哪些文件被编译、如何编译
3. **类型检查**：配置严格的类型检查规则
4. **输出控制**：指定编译目标、模块格式、输出目录等

### 6.1.2 最小配置

```json
{
  "compilerOptions": {}
}
```

这个最小配置会让 TypeScript：
- 编译当前目录及子目录下所有 `.ts` 文件
- 使用默认编译选项
- 输出到当前目录

### 6.1.3 核心配置项详解

#### 编译目标与模块系统

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "moduleResolution": "bundler"
  }
}
```

**target（编译目标）**：
- **概念定义**：指定编译后 JavaScript 的 ECMAScript 版本
- **工作原理**：决定哪些语法需要降级、哪些 API 需要 polyfill

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| `ES3` | ES3（默认） | 兼容 IE8 及以下 |
| `ES5` | ES5 | 兼容 IE11 |
| `ES2015`~`ES2024` | 对应年份的 ES 版本 | 现代浏览器/Node.js |
| `ESNext` | 最新 ES 特性 | 最新环境，实验性特性 |

**module（模块系统）**：
- **概念定义**：指定生成的模块代码格式

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| `CommonJS` | CommonJS 格式 | Node.js 项目 |
| `ESNext` / `ES2015`+ | ES Module 格式 | 浏览器、现代打包工具 |
| `AMD` | RequireJS AMD 格式 | 遗留项目 |
| `UMD` | 通用模块定义 | 库开发 |
| `System` | SystemJS 格式 | 特定场景 |

**lib（库文件）**：
- **概念定义**：指定编译时包含的类型定义文件
- **工作原理**：TypeScript 内置了各种环境的类型定义，通过 lib 选择加载

```json
{
  "compilerOptions": {
    // 浏览器环境
    "lib": ["DOM", "DOM.Iterable", "ES2022"],
    
    // Node.js 环境
    "lib": ["ES2022"],
    
    // Web Worker
    "lib": ["WebWorker", "ES2022"],
    
    // 完整列表
    "lib": [
      "ES5", "ES6", "ES2015", "ES2016", "ES2017", "ES2018", "ES2019",
      "ES2020", "ES2021", "ES2022", "ES2023", "ESNext",
      "DOM", "DOM.Iterable", "WebWorker", "ScriptHost"
    ]
  }
}
```

**moduleResolution（模块解析策略）**：
- **概念定义**：指定 TypeScript 如何解析模块导入路径

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| `classic` | 经典解析（旧版） | 不推荐使用 |
| `node` | Node.js 解析 | 传统 Node.js 项目 |
| `bundler` | 打包工具解析 | Vite、Webpack 等现代工具 |

### 6.1.4 严格模式配置

```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "noImplicitThis": true,
    "alwaysStrict": true,
    "strictBindCallApply": true,
    "strictFunctionTypes": true,
    "strictPropertyInitialization": true,
    "useUnknownInCatchVariables": true
  }
}
```

**strict（总开关）**：
- **概念定义**：启用所有严格类型检查选项
- **工作原理**：相当于同时开启下方所有子选项

**各子选项详解**：

| 选项 | 功能 | 示例 |
|------|------|------|
| `noImplicitAny` | 禁止隐式 any | `function fn(x) {}` ❌ x 会被推断为 any |
| `strictNullChecks` | 严格 null 检查 | `let x: string = null` ❌ |
| `noImplicitThis` | 禁止隐式 this | `function fn() { this.x }` ❌ |
| `alwaysStrict` | 严格模式编译 | 输出 `"use strict"` |
| `strictBindCallApply` | 严格 bind/call/apply 检查 | `fn.call(null, 1)` ✅ 类型检查 |
| `strictFunctionTypes` | 严格函数类型检查 | 禁止双向协变 |
| `strictPropertyInitialization` | 类属性初始化检查 | 类属性必须在构造函数中初始化 |
| `useUnknownInCatchVariables` | catch 变量类型为 unknown | `catch (e)` 中 e 是 unknown |

### 6.1.5 文件包含与排除

```json
{
  "include": [
    "src/**/*",
    "types/**/*.d.ts"
  ],
  "exclude": [
    "node_modules",
    "dist",
    "**/*.test.ts",
    "**/*.spec.ts"
  ],
  "files": [
    "src/index.ts",
    "src/app.ts"
  ]
}
```

**工作原理**：
- `files`：精确指定要编译的文件列表
- `include`：使用 glob 模式包含文件
- `exclude`：从 include 中排除的文件

**优先级**：`files` > `include` - `exclude`

### 6.1.6 输出控制配置

```json
{
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "declarationDir": "./dist/types",
    "declarationMap": true,
    "sourceMap": true,
    "inlineSourceMap": false,
    "removeComments": true,
    "noEmit": false,
    "noEmitOnError": true
  }
}
```

**关键选项说明**：

| 选项 | 功能 | 推荐值 |
|------|------|--------|
| `outDir` | 输出目录 | `./dist` |
| `rootDir` | 源文件根目录 | `./src` |
| `declaration` | 生成 .d.ts 文件 | `true`（库开发） |
| `declarationMap` | 生成 .d.ts.map | `true`（调试用） |
| `sourceMap` | 生成 .js.map | `true`（开发环境） |
| `removeComments` | 移除注释 | `true`（生产） |
| `noEmit` | 不输出文件 | `true`（纯类型检查） |

### 6.1.7 模块解析与路径映射

```json
{
  "compilerOptions": {
    "baseUrl": "./src",
    "paths": {
      "@/*": ["*"],
      "@components/*": ["components/*"],
      "@utils/*": ["utils/*"],
      "@types": ["types/index.ts"]
    },
    "resolveJsonModule": true,
    "allowSyntheticDefaultImports": true,
    "esModuleInterop": true
  }
}
```

**baseUrl + paths**：
- **概念定义**：配置模块路径别名，简化导入语句
- **工作原理**：编译器将别名路径映射到实际文件路径

```typescript
// 没有路径映射
import { Button } from "../../../components/Button";
import { formatDate } from "../../../utils/date";

// 有路径映射
import { Button } from "@/components/Button";
import { formatDate } from "@/utils/date";
```

**esModuleInterop**：
- **概念定义**：允许从没有默认导出的模块中 import
- **工作原理**：为 CommonJS 模块生成辅助代码，支持 default 导入

```typescript
// CommonJS 模块
// lodash.js: module.exports = { debounce: fn, throttle: fn }

// 没有 esModuleInterop
import * as _ from "lodash";
_.debounce(fn);

// 开启 esModuleInterop
import _ from "lodash";
_.debounce(fn);

// 也可以这样
import { debounce } from "lodash";
debounce(fn);
```

## 6.2 类型声明文件（.d.ts）编写指南

### 6.2.1 什么是声明文件？

**概念定义**：声明文件（Declaration File）是以 `.d.ts` 为后缀的 TypeScript 文件，它只包含类型声明，不包含实现代码。

**为什么需要声明文件**：
1. **JavaScript 库的类型支持**：为纯 JS 库添加类型定义
2. **环境声明**：声明全局变量、全局函数等环境信息
3. **模块扩展**：扩展现有模块的类型
4. **类型导出**：将类型定义导出供其他项目使用

### 6.2.2 声明文件基础语法

#### declare 关键字

**概念定义**：`declare` 关键字用于告诉 TypeScript 编译器某个变量、函数、类或模块已经存在于运行时环境中，只需要知道其类型即可。

```typescript
// 声明全局变量
declare const VERSION: string;
declare let debugMode: boolean;
declare var apiBaseUrl: string | undefined;

// 声明全局函数
declare function greet(name: string): string;
declare function setTimeout(handler: () => void, timeout?: number): number;
declare function fetch(url: string, options?: RequestInit): Promise<Response>;

// 声明全局类
declare class Dog {
  name: string;
  constructor(name: string);
  bark(): void;
}

// 声明接口
interface Point {
  x: number;
  y: number;
}

// 声明命名空间
declare namespace Utils {
  function formatDate(date: Date): string;
  function parseJSON<T>(str: string): T;
}
```

#### 模块声明

```typescript
// 声明一个 ES 模块
declare module "my-library" {
  export function doSomething(): void;
  export const version: string;
  export default class MyClass {
    constructor(config: Config);
    run(): Promise<void>;
  }
  
  interface Config {
    debug?: boolean;
    timeout?: number;
  }
}

// 使用
import MyClass from "my-library";
const instance = new MyClass({ debug: true });
```

### 6.2.3 为第三方库编写声明文件

#### 步骤一：分析库的结构

假设我们有一个老旧的 JavaScript 库 `legacy-lib`：

```javascript
// node_modules/legacy-lib/index.js
const calculate = (x, y) => x + y;

const defaultOptions = {
  debug: false,
  timeout: 5000
};

module.exports = {
  calculate,
  defaultOptions
};
```

#### 步骤二：创建声明文件

```typescript
// src/types/legacy-lib.d.ts

// 声明模块
declare module "legacy-lib" {
  // 导出函数
  export function calculate(x: number, y: number): number;
  
  // 导出常量对象
  export const defaultOptions: {
    debug: boolean;
    timeout: number;
  };
}
```

#### 步骤三：在 tsconfig.json 中配置

```json
{
  "compilerOptions": {
    "typeRoots": ["./node_modules/@types", "./types"],
    "types": ["node", "legacy-lib"]
  },
  "include": ["src/**/*", "types/**/*"]
}
```

#### 模块扩展（Module Augmentation）

**概念定义**：模块扩展允许我们在不修改原始模块的情况下，为其添加新的类型、接口或函数声明。

**工作原理**：TypeScript 会合并同名模块的所有声明。

```typescript
// 扩展 lodash 模块
// types/lodash-extension.d.ts

declare module "lodash" {
  // 添加新的函数声明
  export function deepClone<T>(obj: T): T;
  
  // 扩展现有接口
  interface LoDashStatic {
    myCustomUtil(str: string): number;
  }
}

// 使用
import _ from "lodash";
_.myCustomUtil("hello");  // ✅ 类型检查通过
```

#### 扩展 Express

```typescript
// types/express-extension.d.ts

declare global {

    interface Request {
      userId?: string;
      role?: "admin" | "user" | "guest";
    }
    
    interface Response {
      success(data: any): this;
      error(message: string, code?: number): this;
    }
  }
}

export {};
```

### 6.2.5 函数重载声明

```typescript
// 声明函数重载
declare function greet(name: string): string;
declare function greet(names: string[]): string[];
declare function greet(name: string, greeting: string): string;

// 实现（在 JS 文件中）
function greet(nameOrNames: string | string[], greeting = "Hello") {
  if (Array.isArray(nameOrNames)) {
    return nameOrNames.map(n => `${greeting}, ${n}`);
  }
  return `${greeting}, ${nameOrNames}`;
}

// 使用
const single = greet("Alice");              // string
const multiple = greet(["Alice", "Bob"]);   // string[]
const custom = greet("Alice", "Hi");        // string
```

## 6.4 ESLint 集成与代码规范

### 6.4.1 为什么需要 ESLint？

**概念定义**：ESLint 是一个可插拔的 JavaScript/TypeScript 代码检查工具，用于发现和修复代码质量问题。

**为什么需要集成 ESLint**：
1. **代码一致性**：统一团队的代码风格
2. **错误预防**：在编译前发现潜在问题
3. **类型安全增强**：配合 typescript-eslint 进行类型感知检查
4. **自动化修复**：自动修复可修复的问题

### 6.4.2 安装与配置

#### 安装依赖

```bash
# 基础依赖
npm install --save-dev eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin typescript

# 或使用 pnpm
pnpm add -D eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin typescript
```

#### 基础配置（Flat Config，ESLint 9.x+）

```javascript
// eslint.config.js
import tsParser from "@typescript-eslint/parser";
import tseslint from "typescript-eslint";
import eslint from "@eslint/js";

export default tseslint.config(
  // 基础推荐规则
  eslint.configs.recommended,
  ...tseslint.configs.recommended,
  
  // 自定义配置
  {
    languageOptions: {
      parser: tsParser,
      parserOptions: {
        project: "./tsconfig.json",
        tsconfigRootDir: import.meta.dirname
      }
    },
    rules: {
      // 自定义规则
      "@typescript-eslint/no-unused-vars": [
        "error",
        { argsIgnorePattern: "^_", varsIgnorePattern: "^_" }
      ],
      "@typescript-eslint/explicit-function-return-type": [
        "warn",
        { allowExpressions: true }
      ]
    }
  }
);
```

#### 传统配置（.eslintrc.js）


### 6.4.3 核心规则详解

#### 类型相关规则

```javascript
{
  "rules": {
    // 禁止显式 any
    "@typescript-eslint/no-explicit-any": "error",

    "@typescript-eslint/no-non-null-assertion": "warn",
    
    // 要求函数有明确的返回类型
    "@typescript-eslint/explicit-function-return-type": [
      "warn",
      {
        allowExpressions: true,      // 允许表达式
        allowTypedFunctionExpressions: true,  // 允许有类型的函数表达式
        allowHigherOrderFunctions: true  // 允许高阶函数
      }
    ],
    
    // 禁止未使用的变量
    "@typescript-eslint/no-unused-vars": [
      "error",
      {
        argsIgnorePattern: "^_",     // 以_开头的参数忽略
        varsIgnorePattern: "^_",     // 以_开头的变量忽略
        caughtErrorsIgnorePattern: "^_"
```

### 6.4.7 常见问题与解决方案

#### 问题一：ESLint 报错 "Parsing error"

**解决方案**：
```javascript
// 确保 parserOptions 正确配置
parserOptions: {
  project: "./tsconfig.json",
  tsconfigRootDir: __dirname
}
```

#### 问题二：类型检查规则报错

**解决方案**：
```javascript
// 使用 recommended-requiring-type-checking
extends: [
  "plugin:@typescript-eslint/recommended-requiring-type-checking"
]
```

#### 问题三：忽略特定文件的检查

**解决方案**：
```javascript
// eslint.config.js 中添加
{
  ignores: ["**/*.config.js", "**/dist/**", "**/node_modules/**"]
}
```
