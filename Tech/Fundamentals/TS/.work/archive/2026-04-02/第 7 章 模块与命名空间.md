## 7.1 ES Module 支持（import/export）

### 7.1.1 概念定义

**ES Module（ESM）** 是 ECMAScript 2015（ES6）引入的官方模块系统标准，TypeScript 完全支持 ES Module 的 `import` 和 `export` 语法。

**核心概念：**
- **模块（Module）**：任何包含顶级 `import` 或 `export` 声明的 TypeScript 文件都被视为模块
- **导出（Export）**：将模块内部的变量、函数、类、接口等暴露给其他模块使用
- **导入（Import）**：从其他模块中引入已导出的成员供本模块使用
- **作用域隔离**：每个模块拥有独立的作用域，模块内的变量不会污染全局作用域

**为什么需要 ES Module：**
1. **代码组织**：将大型代码库拆分为可维护的小模块
2. **依赖管理**：显式声明模块间的依赖关系
3. **Tree Shaking**：构建工具可以移除未使用的导出，减小打包体积
4. **类型安全**：TypeScript 在编译时检查导入/导出的类型正确性

### 7.1.2 工作原理

**模块解析与编译流程：**

```
TypeScript 源文件 (.ts)
        ↓
   词法/语法分析
        ↓
   模块依赖图构建
        ↓
   类型检查与推断
        ↓
   代码生成 (.js + .d.ts)
```

**导出机制：**

TypeScript 支持两种导出方式：

1. **命名导出（Named Export）**：导出多个具名成员
```typescript
// utils.ts
export const PI = 3.14159;
export function add(a: number, b: number): number {
  return a + b;
}
export interface User {
  id: number;
  name: string;
}
```

2. **默认导出（Default Export）**：每个模块只能有一个默认导出
```typescript
// Logger.ts
export default class Logger {
  log(message: string): void {
    console.log(message);
  }
}
```

**导入机制：**

```typescript
// 命名导入
import { add, User } from './utils';

// 默认导入
import Logger from './Logger';

// 全部导入为命名空间对象
import * as Utils from './utils';

// 重命名导入
import { add as sum } from './utils';

// 仅导入类型（编译后删除，不产生运行时代码）
import type { User } from './utils';
```

### 7.1.3 源码/底层解析

**TypeScript 模块编译输出：**

TypeScript 编译器根据 `tsconfig.json` 中的 `module` 配置，将 ES Module 语法转换为目标模块格式：

```json
{
  "compilerOptions": {
    "module": "ESNext"     // 保持 ES Module 语法
    // 或
    "module": "commonjs",  // 转换为 CommonJS
    // 或
    "module": "NodeNext"   // Node.js 16+ ESM 支持
  }
}
```

**ES Module → CommonJS 转换示例：**

```typescript
// 源文件 (utils.ts)
export const PI = 3.14159;
export function add(a: number, b: number): number {
  return a + b;
}
export default class Calculator {}
```

```javascript
// 编译输出 (utils.js, module: "commonjs")
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.PI = void 0;
exports.add = void 0;

exports.PI = 3.14159;
function add(a, b) {
  return a + b;
}
exports.add = add;

class Calculator {}
exports.default = Calculator;
```

**底层机制解析：**

1. **`__esModule` 标记**：CommonJS 输出中添加 `__esModule: true`，标识这是一个转译后的 ES 模块
2. **`exports` 对象**：命名导出被挂载到 `exports` 对象上
3. **`default` 属性**：默认导出被挂载到 `exports.default`
4. **`import` 转换**：`import` 语句被转换为 `require()` 调用

**TypeScript 类型擦除机制：**

```typescript
// 源文件
export interface Config {
  debug: boolean;
}
export type Status = 'pending' | 'done';
export const config: Config = { debug: true };
```

```javascript
// 编译输出（类型被完全擦除）
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.config = void 0;
exports.config = { debug: true };
```

**类型导入优化（import type）：**

TypeScript 3.8 引入 `import type`，明确标识仅用于类型的导入，编译器确保这些导入不会出现在输出中：

```typescript
// 优化前
import { User } from './types';  // 可能被编译为 require()
const user: User = { id: 1 };

// 优化后
import type { User } from './types';  // 必定被擦除
const user: User = { id: 1 };
```

### 7.1.4 代码示例

**完整模块示例：**

```typescript
// ==================== 文件：src/models/User.ts ====================
// 导出接口（纯类型，编译后删除）
export interface User {
  id: number;
  name: string;
  email: string;
  createdAt: Date;
}

// 导出类型别名
export type UserRole = 'admin' | 'user' | 'guest';

// 导出常量
export const MAX_USERS = 1000;

// 导出函数
export async function fetchUser(id: number): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}

// 导出类（默认导出）
export default class UserService {
  private users: User[] = [];

  async getUser(id: number): Promise<User | undefined> {
    return this.users.find(u => u.id === id);
  }

  async addUser(user: User): Promise<void> {
    if (this.users.length >= MAX_USERS) {
      throw new Error('用户数量已达上限');
    }
    this.users.push(user);
  }
}
```

```typescript
// ==================== 文件：src/services/auth.ts ====================
// 导入类型（使用 import type 优化）
import type { User, UserRole } from '../models/User';
// 导入值和类型
import UserService, { MAX_USERS } from '../models/User';
// 全部导入为命名空间
import * as UserModels from '../models/User';

// 使用导入的成员
const userService = new UserService();

export async function authenticateUser(
  id: number
): Promise<User | null> {
  try {
    const user = await userService.getUser(id);
    return user ?? null;
  } catch (error) {
    console.error('认证失败:', error);
    return null;
  }
}

// 导出类型守卫函数
export function isAdmin(user: User): user is User & { role: UserRole } {
  return 'role' in user && user.role === 'admin';
}
```

```typescript
// ==================== 文件：src/index.ts（聚合导出） ====================
// 重新导出（Re-export）- 不引入到本地作用域
export { default as UserService } from './models/User';
export type { User, UserRole } from './models/User';
export { authenticateUser, isAdmin } from './services/auth';

// 全部重新导出
export * from './models/User';
export * from './services/auth';
```

### 7.1.5 常见误区

**误区 1：混淆默认导出和命名导出**

```typescript
// ❌ 错误：试图用命名导入方式导入默认导出
import { UserService } from './models/User';  // 类型错误！

// ✅ 正确：默认导出使用无花括号导入
import UserService from './models/User';

// ✅ 正确：命名导出使用花括号导入
import { MAX_USERS, fetchUser } from './models/User';
```

**误区 2：忽略 import type 优化**

```typescript
// ❌ 不推荐：类型导入可能被编译为运行时 require()
import { User, Config } from './types';
const user: User = { id: 1 };

// ✅ 推荐：明确标识类型导入，确保编译后删除
import type { User, Config } from './types';
const user: User = { id: 1 };
```

**误区 3：循环依赖问题**

```typescript
// ❌ 文件 A.ts
import { b } from './B';
export const a = 'A';

// ❌ 文件 B.ts
import { a } from './A';
export const b = 'B';
// 可能导致运行时 undefined

// ✅ 解决方案：提取公共依赖到第三方文件
// common.ts
export const a = 'A';
export const b = 'B';

// A.ts
import { b } from './common';
export const a = 'A';

// B.ts
import { a } from './common';
export const b = 'B';
```

**误区 4：在 Node.js ESM 中省略扩展名**

```typescript
// ❌ 错误：Node.js 16+ ESM 要求显式扩展名
import { User } from './User';  // TS2835 错误

// ✅ 正确：使用 .js 扩展名（即使源文件是 .ts）
import { User } from './User.js';

// 配置要求：tsconfig.json 中设置
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}
```

### 7.1.6 最佳实践

**1. 优先使用命名导出**

```typescript
// ✅ 推荐：命名导出更清晰，支持 Tree Shaking
export function helper1() {}
export function helper2() {}
export const CONSTANT = 1;

// ⚠️ 谨慎使用：默认导出不利于 Tree Shaking
export default function main() {}
```

**2. 使用 import type 优化类型导入**

```typescript
// ✅ 类型导入必定被擦除
import type { Config, Options } from './types';

// ✅ 混合导入：值导入和类型导入分开
import { Logger } from './Logger';  // 值
import type { LoggerConfig } from './types';  // 类型
```

**3. 使用路径别名简化导入**

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": "./src",
    "paths": {
      "@models/*": ["models/*"],
      "@services/*": ["services/*"],
      "@utils/*": ["utils/*"]
    }
  }
}
```

```typescript
// ✅ 使用路径别名
import { User } from '@models/User';
import { Logger } from '@services/Logger';
```

**4. 使用聚合导出简化公共 API**

```typescript
// ✅ index.ts 作为模块公共入口
// src/models/index.ts
export type { User } from './User';
export type { Product } from './Product';
export { UserService } from './UserService';

// 使用
import { User, UserService } from '@/models';
```

**5. 避免循环依赖**

- 使用依赖注入模式
- 提取公共类型到独立文件
- 使用事件总线解耦模块

## 7.2 命名空间（Namespace）使用场景

### 7.2.1 概念定义

**命名空间（Namespace）** 是 TypeScript 早期引入的代码组织机制，用于将相关的变量、函数、类、接口等封装在一个命名容器中，避免全局作用域污染。

**核心特性：**
- 使用 `namespace` 关键字定义
- 通过 `export` 控制成员可见性
- 支持嵌套命名空间
- 编译后生成自执行函数（IIFE）和全局对象

**为什么需要命名空间：**
1. **代码组织**：将相关功能分组到逻辑单元中
2. **避免命名冲突**：不同命名空间的同名成员不会冲突
3. **向后兼容**：支持旧版 JavaScript 库的类型定义
4. **声明合并**：支持跨文件的命名空间扩展

### 7.2.2 工作原理

**命名空间定义与使用：**

```typescript
// 定义命名空间
namespace Validation {
  // 内部成员（不导出）
  const lettersRegexp = /^[A-Za-z]+$/;

  // 导出接口
  export interface StringValidator {
    isAcceptable(s: string): boolean;
  }

  // 导出类
  export class LettersOnlyValidator implements StringValidator {
    isAcceptable(s: string): boolean {
      return lettersRegexp.test(s);
    }
  }
}

// 使用命名空间
const validator = new Validation.LettersOnlyValidator();
console.log(validator.isAcceptable("ABC"));  // true
```

**多文件命名空间：**

```typescript
// validation.ts
namespace Validation {
  export interface StringValidator {
    isAcceptable(s: string): boolean;
  }
}

// lettersValidator.ts
/// <reference path="validation.ts" />
namespace Validation {
  const lettersRegexp = /^[A-Za-z]+$/;
  export class LettersOnlyValidator implements StringValidator {
    isAcceptable(s: string) {
      return lettersRegexp.test(s);
    }
  }
}

// numbersValidator.ts
/// <reference path="validation.ts" />
namespace Validation {
  const numbersRegexp = /^[0-9]+$/;
  export class NumbersOnlyValidator implements StringValidator {
    isAcceptable(s: string) {
      return numbersRegexp.test(s);
    }
  }
}
```

### 7.2.3 源码/底层解析

**命名空间编译输出：**

TypeScript 将命名空间编译为 JavaScript 的自执行函数（IIFE）和全局对象：

```typescript
// 源文件
namespace Utility {
  export function log(message: string): void {
    console.log(message);
  }
  export function error(message: string): void {
    console.error(message);
  }
}
```

```javascript
// 编译输出（module: "none"）
var Utility;
(function (Utility) {
  function log(message) {
    console.log(message);
  }
  Utility.log = log;

  function error(message) {
    console.error(message);
  }
  Utility.error = error;
})(Utility || (Utility = {}));
```

**底层机制解析：**

1. **变量声明提升**：`var Utility` 声明被提升到作用域顶部
2. **IIFE 模式**：使用立即执行函数封装内部实现
3. **对象挂载**：导出的成员被挂载到命名空间对象上
4. **合并支持**：`Utility || (Utility = {})` 支持跨文件合并

**嵌套命名空间编译：**

```typescript
// 源文件
namespace MyApp {
  export namespace Validators {
    export class EmailValidator {}
  }
}
```

```javascript
// 编译输出
var MyApp;
(function (MyApp) {
  let Validators;
  (function (Validators) {
    class EmailValidator {}
    Validators.EmailValidator = EmailValidator;
  })(Validators = MyApp.Validators || (MyApp.Validators = {}));
})(MyApp || (MyApp = {}));
```

**命名空间 vs 模块 编译对比：**

```typescript
// 命名空间方式
namespace Math {
  export const PI = 3.14;
  export function add(a: number, b: number) {
    return a + b;
  }
}

// 模块方式
// math.ts
export const PI = 3.14;
export function add(a: number, b: number) {
  return a + b;
}
```

```javascript
// 命名空间编译输出（生成全局对象）
var Math;
(function (Math) {
  Math.PI = 3.14;
  function add(a, b) { return a + b; }
  Math.add = add;
})(Math || (Math = {}));

// 模块编译输出（CommonJS）
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.PI = void 0;
exports.add = void 0;
exports.PI = 3.14;
function add(a, b) { return a + b; }
exports.add = add;
```

### 7.2.4 代码示例

```typescript
  export function truncate(str: string, length: number): string {

    return str.slice(0, length) + '...';
```

### 7.2.5 常见误区

**误区 1：在 2026 年仍然使用命名空间组织业务代码**

```typescript
// ❌ 不推荐：命名空间不利于 Tree Shaking
namespace Validators {
  export function validateEmail(email: string) {}
  export function validatePhone(phone: string) {}
}

// ✅ 推荐：使用 ES Module
// validators.ts
export function validateEmail(email: string) {}
export function validatePhone(phone: string) {}

// 使用
import { validateEmail } from './validators';
```

**误区 2：混淆命名空间和模块的使用场景**

```typescript
// ❌ 错误：在模块文件中使用命名空间
// user.ts (这是一个模块文件)
namespace User {
  export interface User {
    id: number;
    name: string;
  }
}

// ✅ 正确：直接使用模块导出
// user.ts
export interface User {
  id: number;
  name: string;
}
```

**误区 3：忘记使用 export 导出成员**

```typescript
// ❌ 错误：内部成员无法在外部访问
namespace Utils {
  function log(message: string) {  // 未导出
    console.log(message);
  }
}
Utils.log('test');  // 类型错误！

// ✅ 正确：使用 export 导出
namespace Utils {
  export function log(message: string) {
    console.log(message);
  }
}
Utils.log('test');  // 正确
```

**误区 4：忽略命名空间的运行时开销**

```typescript
// ⚠️ 注意：命名空间编译后生成真实的 JavaScript 对象
// 占用运行时内存，增加代码体积

// ❌ 不推荐：大型命名空间增加包体积
namespace HugeLibrary {
  export function func1() { /* ... */ }
  export function func2() { /* ... */ }
  // ... 数百个函数
}

// ✅ 推荐：使用模块 + Tree Shaking
// 构建工具可以移除未使用的导出
```

### 7.2.6 最佳实践

**1. 2026 年命名空间的推荐使用场景**

```typescript
// ✅ 场景 1：声明文件中合并类型定义
declare namespace PIXI {
  export class Sprite {}
  export class Texture {}
}

// ✅ 场景 2：与旧版 JavaScript 库交互
declare namespace LegacyLib {
  export function oldFunction(): void;
}

// ✅ 场景 3：定义全局工具对象（不推荐用于业务代码）
declare namespace Utility {
  function log(msg: string): void;
}

// ❌ 避免：业务代码使用命名空间
// 应使用 ES Module 替代
```

**2. 从命名空间迁移到模块**

```typescript
// 原始命名空间代码
namespace Validation {
  export function validateEmail(email: string) {}
  export function validatePhone(phone: string) {}
}

// 迁移为模块
// validation.ts
export function validateEmail(email: string) {}
export function validatePhone(phone: string) {}

// 使用
import { validateEmail, validatePhone } from './validation';
```

```typescript

namespace Validation {}

// ✅ 使用有意义的名称

// ❌ 避免与全局变量冲突
// namespace Array {}  // 不要覆盖内置对象
```

**4. 跨文件命名空间合并（仅在必要时使用）**

```typescript

// extension.ts
```

## 7.3 混合模式项目迁移（CommonJS + ES Module）

### 7.3.1 概念定义

**混合模式项目** 指同时包含 CommonJS（`require`/`module.exports`）和 ES Module（`import`/`export`）的项目，通常出现在以下场景：

1. **遗留项目迁移**：老项目使用 CommonJS，新功能使用 ES Module
2. **第三方库兼容**：依赖的库同时提供两种模块格式
3. **Node.js 过渡期**：Node.js 14-16 期间的模块兼容
4. **构建工具链**：打包工具（如 Webpack）处理不同模块格式

**为什么需要混合模式支持：**
- 渐进式迁移，降低重构风险
- 兼容不同模块系统的第三方库
- 支持不同构建目标的输出

### 7.3.2 工作原理

**TypeScript 的模块互操作机制：**

```json
{
  "compilerOptions": {
    "esModuleInterop": true,      // 启用 ES 模块互操作
    "allowSyntheticDefaultImports": true,  // 允许默认导入
    "module": "commonjs",         // 输出格式
    "target": "ES2020"
  }
}
```

**esModuleInterop 工作原理：**

```typescript
// 源文件：导入 CommonJS 模块
import express from 'express';  // ES Module 语法

// 编译输出（esModuleInterop: true）
"use strict";
var __importDefault = (this && this.__importDefault) || function (mod) {
  return (mod && mod.__esModule) ? mod : { "default": mod };
};
const express_1 = __importDefault(require("express"));

// 使用
const app = (0, express_1.default)();
```

**__importDefault 辅助函数：**

```javascript
// TypeScript 生成的辅助函数
var __importDefault = (this && this.__importDefault) || function (mod) {
  return (mod && mod.__esModule) ? mod : { "default": mod };
};

// 功能：为 CommonJS 模块添加 default 属性
// 使得 ES Module 的默认导入语法可以正常工作
```

### 7.3.3 源码/底层解析

**CommonJS → ES Module 互操作：**

```typescript
// CommonJS 模块：legacy.js
module.exports = {
  helper: function() { return 'helper'; }
};
module.exports.default = function() { return 'default'; };

// TypeScript 导入
import legacy, { helper } from './legacy';  // ✅ esModuleInterop: true
```

**底层转换逻辑：**

```javascript
// TypeScript 生成的代码
const legacy_1 = __importDefault(require("./legacy"));
const legacy = legacy_1.default;        // 默认导入
const { helper } = legacy_1;            // 命名导入

// __importDefault 逻辑
function __importDefault(mod) {
  // 如果模块已经有 __esModule 标记，直接返回
  if (mod && mod.__esModule) {
    return mod;
  }
  // 否则包装为 { default: mod }
  return { "default": mod };
}
```

**ES Module → CommonJS 互操作：**

```typescript
// ES Module：esm.ts
export default class EsmClass {}
export const value = 42;

// CommonJS 导入（编译后）
const esm_1 = require("./esm");
const EsmClass = esm_1.default;  // 默认导出
const value = esm_1.value;       // 命名导出
```

**Node.js 16+ 的 NodeNext 模式：**

```json
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}
```

**NodeNext 的关键变化：**

1. **文件扩展名敏感**：`.ts`、`.mts`、`.cts` 决定输出格式
2. **package.json 的 type 字段**：`"type": "module"` 启用 ESM
3. **显式扩展名**：相对导入必须包含 `.js` 扩展名

```json
// package.json
{
  "type": "module",           // 项目默认为 ESM
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  }
}
```

## 7.4 模块解析策略（Node vs Classic）

### 7.4.1 概念定义

**模块解析（Module Resolution）** 是 TypeScript 编译器确定 `import` 语句引用的模块文件位置的过程。

**两种解析策略：**

1. **Classic（经典）**：TypeScript 早期的解析策略，现已废弃，仅用于向后兼容
2. **Node（Node.js）**：模拟 Node.js 的模块解析行为，现代项目的标准选择

**为什么需要理解模块解析：**
- 解决"找不到模块"的编译错误
- 优化项目结构和导入路径
- 理解路径别名和 baseUrl 的工作原理
- 处理 monorepo 和复杂项目结构

### 7.4.2 工作原理

**相对导入 vs 非相对导入：**

```typescript
// 相对导入（以 ./、../、/ 开头）
import { User } from './User';           // 同目录
import { Logger } from '../utils/Logger'; // 父目录
import '/mod';                           // 根目录

// 非相对导入（不以特殊前缀开头）
import express from 'express';           // 第三方库
import { User } from '@/models/User';    // 路径别名
```

**Classic 策略解析流程：**

```
相对导入：import { b } from "./moduleB"
  当前文件：/root/src/folder/A.ts

  查找顺序：
  1. /root/src/folder/moduleB.ts
  2. /root/src/folder/moduleB.tsx
  3. /root/src/folder/moduleB.d.ts
  4. /root/src/folder/moduleB/index.ts
  5. /root/src/folder/moduleB/index.tsx
  6. /root/src/folder/moduleB/index.d.ts

非相对导入：import { b } from "moduleB"
  查找顺序：
  1. /root/src/folder/moduleB.ts
  2. /root/src/moduleB.ts
  3. /root/moduleB.ts
  4. /moduleB.ts
  ... 直到根目录
```

**Node 策略解析流程：**

```
相对导入：import { b } from "./moduleB"
  与 Classic 策略相同

非相对导入：import { b } from "moduleB"
  查找顺序：
  1. /root/src/node_modules/moduleB/package.json (types 字段)
  2. /root/src/node_modules/moduleB/index.ts
  3. /root/node_modules/moduleB/package.json
  4. /root/node_modules/moduleB/index.ts
  5. /node_modules/moduleB/package.json
  ... 直到根目录
```

### 7.4.3 源码/底层解析

**TypeScript 模块解析源码位置：**

TypeScript 的模块解析逻辑位于 `src/compiler/moduleNameResolver.ts` 文件中。

**关键解析函数：**

```typescript
// TypeScript 源码（简化版）
function resolveModuleName(
  moduleName: string,
  containingFile: string,
  options: CompilerOptions,
  host: ModuleResolutionHost
): ResolvedModule {

  // 1. 判断是相对导入还是非相对导入
  if (isRelativeImport(moduleName)) {
    return resolveRelativeImport(moduleName, containingFile);
  }

  // 2. 非相对导入：使用 Node 策略
  if (options.moduleResolution === ModuleResolutionKind.Node) {
    return resolveNodeModulesImport(moduleName, containingFile);
  }

  // 3. Classic 策略
  return resolveClassicImport(moduleName, containingFile);
}
```

**扩展名解析顺序：**

```typescript
// 源码中的扩展名枚举
enum Extensions {
  TypeScript = 1 << 0,    // '.ts', '.tsx', '.mts', '.cts'
  JavaScript = 1 << 1,    // '.js', '.jsx', '.mjs', '.cjs'
  Declaration = 1 << 2,   // '.d.ts', '.d.mts', '.d.cts'
  Json = 1 << 3,          // '.json'
}

// 解析顺序：
// 1. .ts
// 2. .tsx
// 3. .d.ts
// 4. .js (ESM 互操作)
// 5. .jsx
// 6. index.ts / index.tsx / index.d.ts
```

**package.json 的 types 字段解析：**

```javascript
// TypeScript 查找 package.json 的优先级
// 1. types 字段（推荐）
{
  "name": "my-lib",
  "types": "./dist/index.d.ts"
}

// 2. typings 字段（旧版别名）
{
  "typings": "./dist/index.d.ts"
}

// 3. main 字段（后备）
{
  "main": "./dist/index.js"  // TypeScript 尝试查找对应的 .d.ts
}
```

**Node16/NodeNext 策略的变化：**

TypeScript 4.7+ 引入的 Node16/NodeNext 策略模拟 Node.js ESM 行为：

```typescript
// NodeNext 策略要求：
// 1. 相对导入必须包含扩展名
import { User } from './User.js';  // ✅

// 2. 文件扩展名决定模块格式
//    .ts → 根据 package.json 的 type 决定
//    .mts → ESM
//    .cts → CommonJS

// 3. package.json 的 exports 字段被尊重
{
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  }
}
```

1. **ES Module**：现代 TypeScript 项目的标准模块格式，支持 Tree Shaking 和类型安全
2. **命名空间**：早期代码组织方式，2026 年主要用于声明文件和向后兼容
3. **混合模式迁移**：从 CommonJS 到 ESM 的渐进式迁移策略
4. **模块解析**：理解 Node 和 Classic 策略的区别，配置路径别名优化项目结构

**核心要点：**
- 新项目统一使用 ES Module + NodeNext 策略
- 避免在业务代码中使用命名空间
- 使用 `import type` 优化类型导入
- 配置路径别名提高代码可读性
- 理解模块解析机制，快速诊断导入问题
