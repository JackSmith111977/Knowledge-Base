## 三、核心要点总结

### 3.1 理解最佳的内容

- **类型剥离 vs tsc --noEmit**：类型剥离只移除类型，不检查；`tsc --noEmit` 只做类型验证，不生成 JS。生产环境两者配合。
- **热点代码定义**：热点 = 执行次数多（计数器超过阈值），与类型稳定无关。类型稳定是优化持续生效的前提。
- **去优化机制**：类型假设失败 → 恢复执行上下文 → 回退到 Ignition → 丢弃优化代码
- **文件 I/O vs 网络 I/O**：文件 I/O 在线程池中同步执行（伪异步），网络 I/O 由 OS epoll 异步处理（真正异步）

### 3.2 理解最深的内容

- **Node.js 启动流程**：main() (C++) → 初始化 V8 → 创建 Isolate → 创建 Environment → LoadEnvironment → SpinEventLoop
- **模块数据流向**：JS API → Bindings (C++ 桥接) → libuv → OS 系统调用
- **GC 算法选择原因**：新生代对象 95% 很快死亡 → Scavenge 高效；老生代存活久 → Mark-Sweep 更划算

### 3.3 一语中的的理解

- **V8 推测优化哲学**：假设类型不变，直接生成最优化代码；只在运行时类型变化时才去优化
- **流式读取 vs 分批调度**：流式处理单个大文件（按节奏消费），分批控制多个文件并发

## 四、薄弱点与错题本

| 编号 | 章节 | 薄弱点 | 错误理解 | 正确理解 |
|------|------|--------|----------|----------|
| 1 | 1.1 | 异步 I/O 真正优势 | "充分利用 CPU" | 让 JS 主线程不阻塞，I/O 由 OS/线程池完成 |
| 2 | 1.4 | CPU 密集型处理方案 | "换成 Go/Rust" | Node.js 内有 Worker Threads、child_process |
| 3 | 2.1 | Node.js 启动流程 | "main() 在 V8 中运行" | main() 是 C++ 原生入口，V8 是被初始化的工具 |
| 4 | 2.1 | 模块数据流向 | 不清楚 Bindings 层 | JS → Bindings → libuv → OS，Bindings 不可少 |
| 5 | 2.3 | 文件 vs 网络 I/O | "pending vs polling" | 文件在线程池同步，网络由 OS epoll 异步 |
| 6 | 2.5 | setTimeout vs setImmediate | 不知道 | 主模块中不确定，I/O 回调内 setImmediate 先 |
| 7 | 2.5 | Worker 内存模型 | "不能共享" | 默认不共享，SharedArrayBuffer 可共享 |
| 8 | 3.1 | 事件循环退出机制 | "进入下一个循环" | 无活跃 handle/request 时直接 break 退出 |
| 9 | 3.3 | 嵌套微任务执行顺序 | AGECDF | A→G→E→C→D→F→B（nextTick 优先级最高） |
| 10 | 3.3 | nextTick 优先级原因 | "特权设计" | nextTick 早于 Promise 存在，不能改变已有行为 |
