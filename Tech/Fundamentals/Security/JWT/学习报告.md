## 错题本

### 第 5 章 — 安全最佳实践（4/5）

**题目：** JWT 存在 LocalStorage 中，AT 有效期 7 天，Payload 包含邮箱和角色，指出安全问题。

**回答偏差：** 将邮箱和角色误判为"敏感信息"。实际上邮箱和角色不属于高敏感数据（可存），真正**绝对禁止**的是密码、身份证号、信用卡号。

**纠正：** JWT 的 Payload 是 Base64URL 编码，任何人都可解码。禁止存储的是**高敏感信息**（密码、身份证、信用卡），业务信息（邮箱、角色、用户名）可酌情存放，遵循最小化原则即可。

### 第 6 章 — 实战应用（3/5）

**题目：** API 网关验证 JWT 后通过 HTTP Header（X-User-ID、X-User-Role）传递信息给后端服务，后端直接使用。有什么安全隐患？

**回答偏差：** 仅想到中间人攻击和 HTTPS 方案，没有意识到更关键的攻击路径——**客户端绕过网关直接伪造 Header** 请求后端服务。

**纠正：** 三层防御：
1. 网络隔离：后端服务只允许内网访问
2. 网关签名透传：用内部密钥对透传信息签名，后端验证签名
3. 清除外部 Header：网关转发前删除客户端传入的 X-User-* Header

### 第 7 章 — 常见误区（3/5）

**题目：** 用户修改密码后，所有之前的 JWT Token 必须立即失效，怎么实现？

**回答偏差：** 只想到双 Token + RT 黑名单方案，没有解决 Access Token 过期前的空窗期问题。

**纠正：** 最佳方案是**版本号标记**：
- 用户表增加 `password_version` 字段
- JWT Payload 携带版本号：`"v": 2`
- 修改密码 → 版本号 +1
- 验证时比对版本号，不一致即拒绝
- 优点：立即生效，不需要黑名单

## 核心要点回顾

1. JWT = 签名（非加密），Payload 可被任何人解码
2. 三段式结构：Header（算法）.Payload（声明）.Signature（防篡改）
3. 7 个标准声明：iss, sub, aud, exp, nbf, iat, jti
4. 双 Token 模式：AT 短命（≤30min）+ RT 长效（7-30d）轮换
5. 前端最佳存储：AT 存内存 + RT 存 HttpOnly Cookie
6. 生产环境推荐 RS256 而非 HS256（公钥可分发）
7. 服务端必须强制算法白名单，禁用 none
8. 微服务场景：网关统一验签，内部服务通过签名透传验证
