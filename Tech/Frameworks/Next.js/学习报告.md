### 二、理解过程

#### 初始理解
学习者在使用 Next.js 前已有一定的 React 基础，对 SSR、SSG 等概念有初步了解。

#### 发现的盲点及解决

| 盲点 | 初始理解 | 正确理解 |
|------|----------|----------|
| **SSG vs ISR** | ISR 是"增量渲染" | ISR 是后台重新生成完整页面后替换 |
| **generateStaticParams** | "跳转时运行" | 构建时执行，预生成静态页面 |
| **error.tsx vs not-found.tsx** | 都是处理 404 | error 处理渲染错误，not-found 处理 404 |
| **HMR vs 刷新** | 都是重新加载 | HMR 保持组件状态，刷新丢失状态 |
| **SSG 部署 vs ISR 更新** | 混淆两者 | SSG 部署是原子替换，ISR 是后台更新 |
| **Server Actions 用途** | 可用于数据获取 | 专用于数据变更（Mutation） |
| **Edge Runtime vs 边缘节点** | 认为是同一概念 | Edge Runtime 是运行环境，边缘节点是物理位置 |

#### 最终理解
通过 8 章学习，学习者能够：
- 准确区分四种渲染模式及其适用场景
- 正确设计 Server/Client Component 的组件结构

- 识别并避免常见的水合错误和性能陷阱

### 四、核心要点总结

#### 1. 四种渲染模式

| 模式 | HTML 生成时机 | 适用场景 |
|------|--------------|----------|
| CSR | 浏览器端 | Dashboard、后台系统 |
| SSR | 每次请求 | 个性化内容、实时数据 |
| SSG | 构建时 | 博客、文档等静态内容 |
| ISR | 构建后按需更新 | 商品列表、新闻等半动态内容 |

#### 2. Server vs Client Component

| 维度 | Server | Client |
|------|--------|--------|
| 执行位置 | 服务端 | 客户端 |
| 数据库访问 | ✅ | ❌ |
| 浏览器 API | ❌ | ✅ |
| 交互能力 | ❌ | ✅ |
| 包体积 | 0 | 有 |
| 指令 | `'use server'` | `'use client'` |

**设计原则：** SC 读，SA 写，CC 负责交互行为

#### 3. 缓存策略

```javascript
// ISR 时间缓存
fetch(url, { next: { revalidate: 3600 } })

// 标签缓存（可批量失效）
fetch(url, { next: { tags: ['products'] } })
revalidateTag('products', 'max')

// 不缓存（实时数据）
fetch(url, { cache: 'no-store' })
```

#### 4. 水合错误避免

- 避免在服务端和客户端生成不同内容（时间戳、随机数）
- 使用 `useEffect` 延迟浏览器特定代码
- 暗黑模式使用内联脚本注入

### 五、待加强内容

1. **SSG vs ISR 的本质区别**
   - SSG：构建时生成，部署时原子替换
   - ISR：后台增量更新，首次请求返回旧内容

2. **HMR 的核心优势**
   - 保持组件状态（表单、计数器、滚动位置）

3. **revalidateTag vs revalidatePath**
   - Tag：跨页面批量失效
   - Path：仅失效指定页面

### 二、本次学习重点

#### 1. Server Actions RPC 架构（新增）

**核心机制：**
- 客户端触发 → 序列化为 POST 请求 → 服务端执行 → RSC Payload 响应
- Action ID 注册表（`__action_manifest__.js`）
- `use server` 指令的编译时转换
- 渐进式增强（无 JS 也能提交表单）

**安全机制：**
- CSRF 保护（allowedOrigins）
- 请求体大小限制（bodySizeLimit）
- 同源验证（Referer + Origin）

#### 2. API Route vs Server Actions 完整对比（新增）

| 维度 | API Route | Server Actions |
|------|-----------|----------------|
| 本质 | RESTful API 端点 | RPC 远程过程调用 |
| 触发来源 | 任何 HTTP 客户端 | 仅限 React 组件内 |
| HTTP 方法 | GET/POST/PUT/DELETE 等 | 仅 POST |
| 响应格式 | 任意（JSON/HTML/Text） | RSC Payload |
| UI 更新 | 手动处理 | 自动更新 |
| 渐进式增强 | ❌ | ✅ |
| 外部调用 | ✅ | ❌ |

#### 3. 流式渲染与渲染模式配合（新增）

**关键理解：流式的前提是 Suspense 边界**

| 渲染模式 | 是否支持流式 | 条件 |
|----------|--------------|------|
| 纯 CSR（无 Suspense） | ❌ | 无 Suspense 边界 |
| CSR + Suspense | ✅ | 有 Suspense 边界 |
| SSR + Suspense | ✅ | 有 Suspense 边界 |
| ISR + Suspense | ✅ | 有 Suspense 边界 |
| SSG + Suspense (PPR) | ✅ | PPR 架构 |

#### 4. 四层缓存机制（复习）

| 层级 | 名称 | 作用域 | 生命周期 |
|------|------|--------|----------|
| L1 | Request Memoization | 单次请求 | 请求结束 |
| L2 | Data Cache | 跨请求 | 直到失效 |
| L3 | Full Route Cache | 完整路由 | 直到失效 |
| L4 | Client Router Cache | 客户端 | 30 秒 TTL |

#### 5. API Route 执行位置理解（重点澄清）

| 调用方 | 请求类型 | 是否经过网络 |
|--------|----------|--------------|
| Client Component → API Route | 客户端请求 | ✅ 是 |
| Server Component → API Route | 服务端请求 | ❌ 否（内部调用） |
| Client Component → 外部 API | ❌ CORS 限制 | - |
| Server Component → 外部 API | ✅ 无限制 | - |

### 四、核心要点总结（新增）

#### 1. Server Actions 调用流程

```
用户点击表单
    ↓
客户端序列化 FormData + $ACTION_ID
    ↓
POST 请求发送到服务端
    ↓
服务端解析 Action ID 并执行
    ↓
返回 RSC Payload
    ↓
客户端更新 UI
```

#### 2. 数据获取策略

| 场景 | 推荐方案 |
|------|----------|
| 获取外部 API 数据 | Server Component 直接 fetch |
| 需要客户端交互 | API Route 代理 + CC fetch |
| 表单提交/Mutation | Server Actions |
| 第三方 Webhook | API Route |

#### 3. 渲染模式选择决策树

```
内容是否需要 SEO？
    ↓ 否 → CSR（Dashboard、工具类）
    ↓ 是
内容是否频繁变化？
    ↓ 否 → SSG（博客、文档）
    ↓ 是
内容是否个性化？
    ↓ 否 → ISR（商品页、新闻）
    ↓ 是 → SSR（个性化内容、用户 Dashboard）
```
