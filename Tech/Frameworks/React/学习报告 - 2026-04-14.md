## 三、核心要点总结

### 3.1 Fiber 架构（理解最佳）

- 递归改循环：调用栈（黑盒）→ 堆内存指针（白盒）
- 双缓冲：Current 树 + WorkInProgress 树 = 空间换时间 + UI 不撕裂
- Reconciliation 可中断，Commit 不可中断（UI 一致性/状态同步/事件监听）
- MessageChannel 延迟最低且不阻塞渲染，优先级是 React 自己实现的

### 3.2 闭包陷阱（理解最深）

- 不是"永远第一次"，而是"锁死触发那一刻的快照"
- 每次渲染 = 重新执行组件函数 = 所有变量重新声明 = 所有函数重新创建
- setTimeout 引用的是"触发那次点击"所在渲染周期的值
- 消除方法：`setCount(prev => prev + 1)` 函数式更新

### 3.3 SSR vs RSC（一语中的）

- SSR 是渲染模式，RSC 是组件类型
- SSR 解决"用户看到得快"，RSC 解决"客户端包要小"
- TTFB 更慢、FCP 更快、TTI 可能更慢

## 四、待加强内容

| 薄弱点 | 当前理解 | 正确理解 |
|--------|---------|---------|
| MessageChannel 机制 | 认为自带优先级 | 本身无优先级，延迟最低 + React 自己实现优先队列 |
| 批量更新 | 认为值一致就合并 | 不管值是什么，queue 中所有更新都会被依次消费 |
| useEffect 执行时机 | 空依赖=每次，无依赖=一次 | 正好反过来：无依赖=每次，空依赖=一次 |
| Provider 嵌套 | 读到外层的值 | useContext 读最近的（内层的）Provider 值 |
| useRef 触发机制 | 浅比较识别新值 | setter 内部调了 scheduleRender()，ref 没调 |
| React.memo 性能开销 | 没有引用 props 时不划算 | 浅比较耗时 > 渲染耗时时不划算 |
