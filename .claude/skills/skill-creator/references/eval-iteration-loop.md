# Eval 迭代循环

> 官方的"写 → 测 → 评 → 改"迭代机制，适配本地 Skill 创建流程
>
> **版本：** 1.0.0 | **来源：** anthropics/skills/skill-creator 核心循环

---

## 核心循环

```
明确目标 → 起草/编辑 Skill → 对测试 prompt 执行 → 与用户一起评估结果 → 根据反馈改进 → 重复直到满意
```

---

## 阶段 A：定义测试用例

**时机：** 阶段 3（用例定义）之后，阶段 5（生成草稿）之前

### A1：编写 evals.json

基于阶段 3 确认的 2-3 个用例，创建 `evals/evals.json`：

```json
{
  "skill_name": "my-skill-name",
  "evals": [
    {
      "id": "eval-001",
      "prompt": "[触发词 1] [测试内容]",
      "expected_output": "[预期输出描述]",
      "expectations": [
        "期望 1 — 具体可验证",
        "期望 2 — 具体可验证",
        "期望 3 — 具体可验证"
      ]
    }
  ]
}
```

### A2：断言编写原则

- **要具体** — "输出包含 X 章节"，不是"输出质量好"
- **要可区分** — 好的断言在 Skill 真正成功时通过，失败时不通过
- **要覆盖** — 正常路径 + 边界情况 + 错误输入
- **先写 prompt，后写期望** — 不要一开始就写断言

### A3：生成触发词评估集（用于描述优化）

创建 20 条评估查询：
- **10 条应触发** — 正常应该触发 Skill 的输入
- **10 条不应触发** — 不应该触发 Skill 的输入

用于验证 `triggers` 和 `description` 的准确性。

---

## 阶段 B：执行测试

**时机：** 阶段 6（写入文件）之后

### B1：运行测试用例

对每个测试用例：

1. 使用测试 prompt 触发 Skill
2. 记录完整的执行转录（工具调用、步骤、输出）
3. 保存所有输出文件到 `evals/runs/<timestamp>/outputs/`
4. 记录工具调用次数到 `evals/runs/<timestamp>/outputs/metrics.json`
5. 记录时间数据到 `evals/runs/<timestamp>/timing.json`

### B2：生成用户备注

执行完成后，在 `evals/runs/<timestamp>/outputs/user_notes.md` 中记录：
- **不确定项** — 不确定是否正确的地方
- **需要人工审核** — 需要人类判断的地方
- **变通方案** — Skill 未按预期工作时使用的替代方案

---

## 阶段 C：评分

**时机：** 阶段 B 之后

### C1：加载 Grader

加载 `references/grader-agent.md`，执行评分流程：

1. 读取转录文件
2. 检查输出文件
3. 逐一评估每条期望（PASS/FAIL + 证据）
4. 提取并验证隐含声明
5. 审查评估本身的质量
6. 输出 `grading.json`

### C2：汇总结果

```markdown
## 评估结果摘要

| 测试用例 | PASS | FAIL | 通过率 | 耗时 |
|----------|------|------|--------|------|
| eval-001 | 3    | 1    | 75%    | 165s |
| eval-002 | 4    | 0    | 100%   | 142s |
| eval-003 | 2    | 2    | 50%    | 198s |
| **总计** | **9** | **3** | **75%** | **505s** |
```

### C3：展示给用户

展示评分结果，重点突出：
- **失败的期望** — 哪些功能未达标
- **隐含声明的问题** — Grader 发现的未在期望中但实际存在的问题
- **评估建议** — Grader 对评估本身质量的反馈
- **用户备注汇总** — 执行者的不确定项

---

## 阶段 D：改进 Skill

**时机：** 阶段 C 之后，用户确认需要改进

### D1：从反馈中泛化

1. **分析失败期望** — 为什么会失败？是指令不清楚还是步骤缺失？
2. **分析 Grader 建议** — 评估本身是否需要改进？
3. **分析用户备注** — 执行过程中遇到了什么问题？

### D2：应用改进

改进方向：

| 问题类型 | 改进方式 |
|----------|----------|
| 指令不清 | 重写具体指令，加入 DO NOT |
| 步骤缺失 | 补充缺失步骤到 Pipeline |
| 输出不符 | 加强 Generator 模板或 Style Guide |
| 触发不准 | 优化 description 或 triggers |
| 评估过松 | 加强期望断言，检查内容而非表面 |

### D3：保持精简

"Keep the prompt lean." — 改进时优先删除冗余，而不是堆积指令。

### D4：解释 Why

"The why behind a rule helps the agent generalize." — 指令中保留"为什么"的解释，帮助 Agent 理解而非机械执行。

### D5：跨用例模式

"Look for repeated work across test cases." — 如果多个用例都遇到同样问题，把它抽象为通用规则。

---

## 阶段 E：迭代

### E1：更新版本历史

更新 `evals/history.json`：

```json
{
  "iterations": [
    {
      "version": "v2",
      "parent": "v1",
      "expectation_pass_rate": 0.75,
      "changes": "增加网络搜索步骤，优化触发词"
    }
  ]
}
```

### E2：重新运行

重复 阶段 B → C → D，直到：
- 通过率达到可接受水平（建议 ≥ 80%）
- 用户对输出质量满意
- 没有重要的未覆盖结果

### E3：可选 — 盲对比

对于需要严谨验证的场景：

1. 对有 Skill 和无 Skill 分别运行相同测试
2. 由 Grader 对两者分别评分
3. 对比通过率、耗时、Token 消耗
4. 输出 `benchmark.json` 报告改进幅度

## 简化执行指南

由于本地环境可能没有官方的自动化脚本，以下是**简化执行方式**：

### 最小可行流程

1. **写 3 个测试 prompt** — 覆盖正常、边界、错误
2. **对每个 prompt 写 2-4 条期望** — 具体可验证
3. **手动执行** — 逐个 prompt 触发，观察输出
4. **手动评分** — 对照期望逐一 PASS/FAIL
5. **记录结果** — 手动填写 grading.json
6. **改进** — 根据失败项修改 SKILL.md
7. **重测** — 再次执行，直到满意

### 进阶流程（有 SubAgent 时）

1. 对有/无 Skill 版本同时运行（盲对比）
2. 使用 SubAgent 做 Grader 评分
3. 自动化 metrics 和 timing 记录
4. 生成 benchmark 报告
