# Parallel-Task Skill

## 用途

本 Skill 用于启动多个并发 SubAgent 执行通用并发任务，根据任务特性自动选择**批量并行模式**或**任务队列模式**。

### 适用场景

| 场景类型 | 说明 | 执行模式 |
|----------|------|----------|
| **批量代码审查** | 同时审查多个模块/文件的代码质量 | 批量并行 |
| **多维度分析** | 安全/性能/规范/测试覆盖率等专项分析 | 批量并行 |
| **大型重构任务** | JS→TS 迁移等涉及数百文件的批量处理 | 任务队列 |
| **通用并发任务** | 用户自定义任务列表 | 自动选择 |

### 不适用场景

- 需要 SubAgent 之间直接沟通协作的任务 → 使用 Agent Teams
- 单个简单任务 → 直接执行，无需并发
- 任务之间存在强依赖关系且无法并行 → 顺序执行

## 工作流程

### 阶段 1：任务接收与分析

**步骤 1.1：接收用户输入**

用户输入格式：
```
[触发词] + [任务描述] + [可选：执行范围/文件列表]
```

示例：
```
并发执行以下任务：
1. 审查 src/components 的代码规范
2. 检查 src/utils 的安全漏洞
3. 分析 src/api 的性能问题
```

**步骤 1.2：任务拆解与分类**

分析用户输入，拆解为独立子任务：

```markdown
## 任务分析结果

**任务总数：** N 个
**可并行任务：** M 个（无依赖关系）
**需顺序任务：** K 个（存在依赖关系）

### 任务清单
1. 任务 A - [类型] - [执行模式]
2. 任务 B - [类型] - [执行模式]
3. ...
```

**步骤 1.3：模式选择决策**

```
┌─────────────────────────────────────┐
│      任务是否可以并行执行？          │
└─────────────────────────────────────┘
                │
        ┌───────┴───────┐
        ▼               ▼
    是：批量并行    否：任务队列
```

| 判断条件 | 执行模式 | 说明 |
|----------|----------|------|
| 所有任务独立，无依赖关系 | 批量并行 | 同时启动多个 SubAgent |
| 任务之间存在依赖关系 | 任务队列 | 按顺序依次执行 |
| 混合情况 | 混合模式 | 部分并行 + 部分顺序 |

---

### 阶段 2：批量并行模式

**步骤 2.1：SubAgent 委派策略**

根据任务类型自动委派给专属 SubAgent：

| 任务类型 | 委派目标 | 说明 |
|----------|----------|------|
| 代码审查 | `code-reviewer` | 检查代码规范、潜在 Bug |
| 安全检查 | `security-scanner` | 扫描 SQL 注入、XSS、CSRF |
| 性能分析 | `performance-analyzer` | 检测 N+1 查询、内存泄漏 |
| 测试生成 | `test-generator` | 生成单元测试和集成测试 |
| 通用任务 | `general-purpose` | 复杂、多步骤任务 |

**步骤 2.2：并发执行**

```markdown
## 执行状态

**并发 SubAgent 数量：** N
**预计执行时间：** ~2-5 分钟/任务

### 进度追踪
- [x] SubAgent 1: [任务描述] - 执行中
- [x] SubAgent 2: [任务描述] - 执行中
- [ ] SubAgent 3: [任务描述] - 等待中
- ...
```

**步骤 2.3：结果收集**

每个 SubAgent 完成后，收集结果摘要：

```markdown
## SubAgent 结果摘要

### SubAgent 1: [名称]
**状态：** 完成
**耗时：** X 分钟
**关键发现：**
- 发现 1
- 发现 2
**输出文件：** [文件路径]
```

---

### 阶段 3：任务队列模式

**步骤 3.1：依赖关系分析**

```markdown
## 任务依赖关系

任务 A → 任务 B → 任务 C
  │
  └──→ 任务 D
```

**步骤 3.2：顺序执行**

```
执行流程：
1. 执行任务 A（无依赖）
2. 等待任务 A 完成
3. 执行任务 B（依赖 A）
4. 执行任务 C（依赖 B）
5. 执行任务 D（依赖 A）
```

**步骤 3.3：进度追踪**

```markdown
## 执行进度

**当前阶段：** 3/5
**进行中：** 任务 C
**已完成：** 任务 A、任务 B
**等待中：** 任务 D、任务 E
```

---

### 阶段 4：结果整合与输出

**步骤 4.1：分 Agent 报告生成**

每个 SubAgent 输出独立章节：

```markdown
# [任务名称] 执行报告

## 执行概览

**执行模式：** [批量并行/任务队列]
**SubAgent 数量：** N
**总耗时：** X 分钟
**完成时间：** YYYY-MM-DD HH:mm

---

## SubAgent 报告

### 报告 1: [SubAgent 名称/任务类型]

**执行时间：** X 分钟
**任务描述：** [具体任务]
**执行范围：** [文件列表/目录]

#### 发现汇总
- 关键发现 1
- 关键发现 2

#### 详细结果
[具体内容]

#### 建议与后续
- 建议 1
- 建议 2

---

### 报告 2: [SubAgent 名称/任务类型]
...
```

**步骤 4.2：整体汇总**

```markdown
## 整体汇总

### 关键指标
| 指标 | 数值 |
|------|------|
| 总任务数 | N |
| 完成数 | M |
| 发现问题数 | K |
| 总耗时 | X 分钟 |

### 优先级行动项
1. **Critical:** [需要立即处理的问题]
2. **Warning:** [需要注意的问题]
3. **Info:** [改进建议]
```

---

## 输出格式规范

### 报告结构

```markdown
# [任务名称] 执行报告

## 执行概览 → 整体情况

## SubAgent 报告 → 分 Agent 报告（每个 SubAgent 独立章节）
### 报告 1: [名称]
### 报告 2: [名称]
...

## 整体汇总 → 整合发现和建议
```

### 输出内容要求

| 要素 | 必须包含 | 说明 |
|------|----------|------|
| 执行模式 | ✅ | 批量并行/任务队列 |
| SubAgent 数量 | ✅ | 并发执行的子代理数量 |
| 分 Agent 报告 | ✅ | 每个 SubAgent 独立章节 |
| 任务进度 | ✅ | 执行中实时更新 |
| 关键发现 | ✅ | 汇总各 SubAgent 的核心发现 |
| 后续建议 | ✅ | 可操作的改进步骤 |

## 注意事项

### 并发限制

| 限制类型 | 数值 | 说明 |
|----------|------|------|
| 推荐并行数 | ~10 个 | 通过任务队列协调 |
| 最大并发数 | 49+ | 某些场景下支持 |
| 连续运行时间 | 2.5 小时 + | 长时间任务支持 |

### 文件冲突避免

- 尽量分配不同文件给不同 SubAgent
- 避免多个 SubAgent 同时编辑同一文件
- 只读任务（审查/分析）可安全并行

### 主上下文保护

- SubAgent 只返回结果摘要，不返回详细推理过程
- 主上下文保持干净，只保留决策信息
- 复杂探索任务全部外包给 SubAgent

### Token 效率

| 模式 | Token 消耗 | 说明 |
|------|------------|------|
| 批量并行 | ~30-40K | 比单一 Agent 节省 30-40% |
| 任务队列 | ~25-35K | 顺序执行，上下文复用 |

## 使用示例

### 示例 1：批量代码审查

```
用户：并发执行以下审查任务
1. 审查 src/components 的代码规范
2. 检查 src/utils 的安全漏洞
3. 分析 src/api 的性能问题

Skill: 启动 3 个 SubAgent 并行执行
- SubAgent 1 → 代码规范审查
- SubAgent 2 → 安全漏洞扫描
- SubAgent 3 → 性能问题分析

输出：分 Agent 报告 + 整体汇总
```

### 示例 2：全面项目分析

```
用户：全面分析这个项目

Skill: 自动拆解为多个专项分析
- 安全漏洞扫描（SQL 注入、XSS、CSRF）
- 性能问题检测（N+1 查询、内存泄漏）
- 代码规范检查（命名、格式、注释）
- 测试覆盖率分析
- 依赖安全检查

输出：5 份专项报告 + 综合建议
```

### 示例 3：大型重构任务

```
用户：将项目从 JS 迁移到 TS

Skill: 识别依赖关系，采用任务队列模式
1. 分析项目结构（前置任务）
2. 生成类型定义（依赖 1）
3. 迁移前端文件（依赖 2）
4. 迁移后端文件（依赖 2）
5. 类型验证（依赖 3、4）

输出：分阶段报告 + 迁移进度追踪
```

- [ ] 选择合适的 SubAgent 类型

- [ ] 实时更新任务进度
- [ ] 监控 SubAgent 执行状态

### 执行后检查

- [ ] 收集所有 SubAgent 结果
- [ ] 生成分 Agent 报告
- [ ] 输出整体汇总和建议
- [ ] 更新进度追踪文件
