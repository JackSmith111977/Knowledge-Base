# 工作流程细节

> 本文档描述 project-migration Skill 各阶段的详细执行步骤

---

## 阶段 0：启动与工作区检测

### 步骤 0.1：欢迎与范围说明

```markdown
🚀 **项目迁移向导**

我将通过三阶段流程为你的老项目建立文档优先 AI 开发系统：
1. 现状分析：扫描项目结构、识别技术栈、分析现有文档
2. 无限询问：动态询问项目配置、文档偏好、迁移优先级（不固定问题数量）
3. 文档生成：生成完整的.claude/和 docs/文档体系

**支持的工作区类型：**
- 单应用项目
- Monorepo（apps/、packages/）
- Trae 工作区（.trae/）
- VSCode 工作区（.code-workspace）

让我们开始。
```

### 步骤 0.2：工作区检测

扫描当前工作目录，识别：

| 工作区类型 | 识别标志 |
|------------|----------|
| **Trae 工作区** | `.trae/` 目录、工作区配置文件 |
| **VSCode 工作区** | `.code-workspace` 文件 |
| **Monorepo** | `apps/`、`packages/` 目录 |
| **单应用** | 标准项目结构（`src/`、`package.json`） |

```markdown
## 工作区检测结果

**检测到的工作区类型：** [Trae/VSCode/Monorepo/单应用]

**发现的应用/目录：**
- [应用 1]：[路径]
- [应用 2]：[路径]

**工作区配置：**
- [从工作区配置文件中读取的根目录列表]

**请选择：**
- [x] 分析所有检测到的应用
- [ ] 选择特定应用（请指定）
- [ ] 手动添加另一个应用的根目录地址

请确认或提供另一个应用的根目录路径。
```

## 阶段 1：现状分析（Status Check）

### 步骤 1.1：项目扫描

自动执行以下分析：

| 分析项 | 说明 |
|--------|------|
| 目录结构 | 生成完整的目录树 |
| 技术栈识别 | 通过 package.json/pom.xml/requirements.txt 等识别 |
| 代码规模 | 统计文件数、代码行数（按语言分类） |
| 现有文档 | 检查 README、设计文档、API 文档等 |
| 依赖关系 | 分析应用间依赖（Monorepo 场景） |
| 配置文件 | 识别构建配置、部署配置等 |
| 工作区配置 | 读取 Trae/VSCode 工作区配置文件 |

### 步骤 1.2：生成现状报告

输出到 `docs/migration/current-status.md`，详见 `templates/assessment-template.md`。

### 步骤 1.3：人类审查确认

```markdown
## 阶段 1 完成：现状报告已生成

**输出文件：** `docs/migration/current-status.md`

### 关键发现摘要
- 识别的技术栈：[摘要]
- 文档完整性：[X]/8 分
- 推荐迁移策略：[策略名称]

请确认：
- 回复「是」继续进入阶段 2（无限询问）
- 回复「否」指出需要修正的内容
- 回复「跳过」暂不处理直接进入阶段 3
```

## 阶段 2：无限询问（Confirmation）

### 步骤 2.1：说明询问流程

```markdown
## 阶段 2：无限询问

我将通过一系列问题了解项目的更多信息，以便生成最合适的文档体系。

**询问原则：**
- 不固定问题数量，直到所有前提明确
- 每个问题后确认你的回答
- 你随时可以说"够了"、"继续"或"跳过"

预计需要 [5-15] 轮询问，取决于项目复杂度。

让我们开始。
```

### 步骤 2.2：核心问题（必问）

#### 问题 1：项目核心目标

```markdown
### 问题 1：项目核心目标

请用一句话描述项目的核心目标：
- 项目要解决什么问题
- 目标用户是谁
- 项目的核心价值

[等待用户回答]

**我的理解：** [复述用户的回答]
是否正确？如有需要补充的请说明。
```

#### 问题 2：团队配置

```markdown
### 问题 2：团队配置

请选择合适的选项：

**团队规模：**
- [ ] 个人项目
- [ ] 小团队（2-10 人）
- [ ] 企业项目（10 人+）

**团队角色：**
- [ ] 全栈开发（前后端都写）
- [ ] 前后端分离（有明确分工）
- [ ] 有专门的产品/设计

[等待用户回答]
```

#### 问题 3：文档详细程度偏好

```markdown
### 问题 3：文档详细程度

请选择合适的详细程度：

**精简版**
- 生成：CLAUDE.md + TECH_STACK.md + migration-plan.md
- 适合：个人项目、快速原型

**标准版**
- 生成：精简版 + PRD.md + APP_FLOW.md + 现状报告
- 适合：小团队、中等复杂度项目

**完整版**
- 生成：标准版 + FRONTEND_GUIDELINES.md + BACKEND_STRUCTURE.md + IMPLEMENTATION_PLAN.md + specs/
- 适合：企业项目、长期维护项目

你选择哪个版本？
```

### 步骤 2.3：条件问题（按需追问）

```markdown
**追问逻辑：**

IF 团队规模 == "小团队" OR "企业项目" THEN
    追问：团队角色分工
    追问：代码审查流程
    追问：团队协作规范
END IF

IF 多应用 == true THEN
    追问：多应用文档组织方式（统一/分离/混合）
    追问：应用间依赖管理策略
END IF

IF 迁移时间约束 == "有截止日期" THEN
    追问：优先级调整
    追问：最小可用范围
END IF

IF 检测到技术债务高 THEN
    追问：技术债务处理策略（重构/保留/替换）
END IF

IF 现有文档 == "有部分文档" THEN
    追问：现有文档是否需要保留
    追问：是否有文档需要更新
END IF

IF 用户回答模糊或不完整 THEN
    追问澄清问题，直到明确
END IF
```

### 步骤 2.4：其他配置问题

#### Agent 对话规约偏好

```markdown
### 对话规约配置

**对话模式偏好：**
- [ ] 简洁直接（只给关键信息）
- [ ] 详细解释（包含原理和背景）
- [ ] 分步骤引导（一步一步来）

**代码生成约束：**
- [ ] 先生成接口/类型，再实现逻辑
- [ ] 优先关注性能
- [ ] 优先关注安全性
- [ ] 优先关注可读性

**审查重点：**
- [ ] 安全性优先
- [ ] 性能优先
- [ ] 规范优先
```

#### 其他重要约定

```markdown
### 其他重要约定

**是否有以下特定需求？**
- [ ] 需要配置 MCP 服务
- [ ] 需要预定义子代理
- [ ] 需要模块化规则文件
- [ ] 有其他团队内部规范

请说明具体需求。
```

### 步骤 2.5：结束条件

```markdown
**询问结束条件（满足任一即可）：**

1. 所有核心问题已回答
2. 用户主动说"够了"、"继续"
3. 连续 2 个问题用户回答"跳过"
4. 已收集足够生成文档的信息
```
