## 8. 知识库文档构建规范

### 8.1 内容深度要求

**每个核心概念必须包含：**
1. **概念定义**：用简洁清晰的语言解释"是什么"和"为什么"
2. **工作原理**：说明内部机制和运作流程，避免只给代码示例
3. **源码/底层解析**：关键特性需附带源码级别的解析或实现思路
4. **代码示例**：配合文字说明的完整代码示例
5. **常见误区**：指出开发者容易误解的地方
6. **最佳实践**：给出实际开发中的推荐使用方式

### 8.2 禁止事项

- 不要只给代码示例而不解释原理
- 不要只罗列 API 表面用法而不深入底层机制
- 不要遗漏关键概念的来龙去脉（如：为什么需要这个特性）
- 不要使用"魔法"等模糊词汇，必须解释清楚背后的实现

### 8.3 官方文档参考

- 必须结合官方文档（如 react.dev）进行内容验证
- 引用官方文档中的关键解释和最佳实践
- 标注重要概念的官方文档链接供深度阅读

### 8.4 源码解析要求

对于以下核心主题，必须包含源码级别的解析：
- 虚拟 DOM 的结构与 diff 算法原理
- Fiber 架构的设计思想与实现
- Hooks 的内部实现机制（useState、useEffect 等）
- 事件系统与合成事件
- 并发模式与调度器原理

### 8.5 示例质量要求

```markdown
❌ 不好的示例（只有代码，没有解释）：
const [count, setCount] = useState(0);

✅ 好的示例（概念 + 原理 + 代码 + 注意事项）：
**原理**：useState 内部维护一个状态链表，每个 Hook 按调用顺序存储在链表中...
**代码**：const [count, setCount] = useState(0);
**注意**：setCount 是异步的，多次调用会合并更新...
```

## 9. 文档生成后检查清单

在写入知识库文档前，必须完成以下检查：

### 9.1 结构检查

- [ ] **章节编号连续且无重复**：使用 `grep "^## \\d+\\." filename.md` 验证
- [ ] **无重复的章节标题**：使用 `grep "^## " filename.md | sort | uniq -d` 检查
- [ ] **目录与实际章节对应**：目录中的每个条目必须有对应的实际章节
- [ ] **章节顺序合理**：子章节编号必须在父章节之后（如 2.1, 2.2, 2.3...）

### 9.2 内容深度检查

- [ ] **每个核心概念包含定义**：解释"是什么"和"为什么"
- [ ] **每个核心概念包含工作原理**：说明内部机制，不只给代码示例
- [ ] **关键特性包含源码解析**：Hooks、Fiber、Diff 算法等必须有底层实现说明
- [ ] **代码示例配有文字说明**：不要只给代码，必须有解释性文字
- [ ] **指出常见误区**：开发者容易误解的地方需要标注

### 9.3 格式检查

- [ ] **Markdown 格式正确**：代码块、表格、列表格式规范
- [ ] **章节分隔符统一**：使用 `---` 分隔主要章节
- [ ] **代码块标注语言**：```jsx、```javascript、```css 等
- [ ] **表格有表头分隔线**：`|---|---|---|`

### 9.4 检查流程示例

```bash
# 1. 检查章节编号
grep "^## " React 核心知识体系.md

# 2. 检查重复标题
grep "^## " React 核心知识体系.md | sort | uniq -d

# 3. 验证目录链接
# 手动比對目录与实际章节是否一一对应
```

### 9.5 发现问题时的处理

- 发现重复章节 → 删除重复内容，保留更完整的版本
- 发现章节错乱 → 重新排序，确保编号连续
- 发现内容过浅 → 补充原理讲解和源码分析
- 发现缺少误区 → 添加常见错误示例和正确做法对比

## 6. Git 提交规范

### 6.1 提交消息格式

- 类型 (scope): 主题
- 类型：`feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`

### 6.2 任务进度追踪

**任务开始前：**
1. 读取 `progress.txt`
2. 更新"进行中"字段，添加当前任务
3. 确认更新后再开始任务

**任务完成后：**
1. 更新 `progress.txt`：
   - 将任务移至"已完成"
   - 添加新的"进行中"任务（如有）
2. 执行 `git add progress.txt` 及相关变更
3. 提交并 push 本次更改

### 6.3 提交前检查

- [ ] `progress.txt` 已更新
- [ ] 提交消息符合规范
- [ ] 相关文档已更新
